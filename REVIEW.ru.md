## 1. Краткое резюме

Hydraline - это хорошо спроектированный, дисциплинированный кодовый базис,
который действительно близок к продакшн-качеству для своей заявленной ниши. Он
решает реальную и актуальную проблему (Flutter Web рендерит в `<canvas>`, поэтому
краулеры и социальные боты видят пустую оболочку). Архитектура - чистая Dart-модель
документа, которая одновременно и стримится с сервера, и компилируется в статический
HTML, с опциональными Flutter-«островами» - продумана, а её защищённость
(типобезопасный `SafeUrl`, контекстно-зависимое экранирование, XSS-фаззинг на
миллион входов) выше среднего для проекта до версии 1.0.

Все блокирующие гейты зелёные на свежем прогоне:

| Гейт | Результат |
|---|---|
| `melos run analyze` (`--fatal-infos --fatal-warnings`) | Проблем не найдено |
| `melos run format:check` | 105 файлов, 0 изменено |
| Тесты ядра (`dart test`) | 212 пройдено |
| Тесты сервера | все пройдено (~93) |
| Тесты Flutter (`flutter test`) | 133 пройдено |
| `melos run boundaries` (правила импортов I1) | OK, запрещённых импортов нет |
| `melos run audit` (сборка SSG + SEO-аудит примера) | 0 ошибок, 0 предупреждений |

Кодовый базис небольшой и честный: ~5.5k строк продакшн-Dart против ~5.4k строк
тестов (примерно 1:1), чистые границы пакетов, зафиксированные тулингом, и
документация, которая в основном соответствует коду.

Основные оговорки касаются не гигиены кода, а зрелости и честности верхнеуровневых
маркетинговых заявлений:

1. **Flutter-рантайм островов L2 не проверен в реальном браузере.** Все тесты -
   это Dart-side widget/unit тесты. В репозитории нет e2e/браузерного стенда
   (репозиторий сам это признаёт: «E2E (Playwright) в репозитории нет»). Самая
   новаторская и хрупкая часть продукта (мульти-view движок, хостящий N островов,
   JS-диспетчер, service worker, гидратация через Declarative Shadow DOM)
   валидируется только через ассерты на Dart-строках, никогда - загрузкой
   страницы.
2. **Реальный семантический пробел в surface Flutter-виджетов**: `Seo.section` и
   `Seo.list` молча отбрасывают свою структурную роль и выдают плоские параграфы
   (детали в 3.1). Чистое Dart-ядро делает всё правильно; Flutter-surface - нет.
3. Некоторые **«автоматические» поведения, разрекламированные в README,
   на практике условны** (bot-aware транспорт отключается, когда сконфигурирован
   кэш; `DocumentBuilder` никогда не получает данные маршрута).

Общая оценка: **B+ / крепкая до-1.0.** Полезность и актуальность высокие; ядро
заслуживает доверия; слой Flutter-островов требует валидации в реальном браузере
до любых заявлений о «продакшене», а горсть поверхностных несостыковок стоит
закрыть.

## 2. Утилитарность, полезность, актуальность

- **Актуальность проблемы: высокая и текущая.** SEO для Flutter Web - реальная,
  нерешённая боль в 2026. Подход «настоящий HTML в первом ответе + гидратация
  островов» - правильный архитектурный ответ, а анти-клоакинг-позиция (билдеры не
  могут видеть `User-Agent`) - реальное отличие.
- **Качество дизайна: высокое.** Единая модель документа, питающая SSR и SSG;
  жёсткие границы пакетов (ядро - чистый Dart, без `flutter`/`dart:ui`/`dart:html`),
  зафиксированные `tool/check_boundaries.dart` и CI. Sealed-иерархия узлов с
  однопроходным сериализатором. Это идиоматично, тестируемо и поддерживаемо.
- **Безопасность-по-построению: сильная.** У `SafeUrl` нет публичного
  конструктора; URL обязаны пройти allowlist схем при построении. Два раздельных
  экранировщика (текст vs атрибут) предотвращают путаницу контекстов. JSON-LD
  экранируется как `\uXXXX`, чтобы предотвратить выход из `</script>`. Подкреплено
  фаззинг-сьютом на 1e6 входов, который проходит.
- **Зрелость: ранняя (0.0.3).** Чистое Dart-ядро (модель документа, сериализатор,
  SEO-артефакты, серверная middleware) правдоподобно для продакшена уже сегодня.
  Flutter-рантайм островов амбициозен и интересен, но фактически не проверен
  end-to-end. Считайте L0/L1 (статика + vanilla/HTMX) готовыми; считайте L2
  (Flutter-острова) бетой.
- **Bus factor / footprint:** маленький, сфокусированный, один автор. Это нормально
  для такой области, но новаторский JS-рантайм лишён браузерного тестового
  покрытия, которого требует его сложность.

Рекомендация для внедряющих: **используйте сегодня для document/hybrid страниц на
L0/L1; пилотируйте L2 Flutter-острова за реальным браузерным тестированием, прежде
чем катить в прод.**

## 3. Подтверждённые находки (проверены по коду)

Легенда серьёзности: CRITICAL (потеря данных / безопасность) · MAJOR (неверное
поведение или вводящее в заблуждение ключевое обещание) · MINOR (краевой случай /
устойчивость) · NIT (косметика).

### 3.1 MAJOR - `Seo.section` / `Seo.list` отбрасывают свою семантическую роль

`packages/hydraline_flutter/lib/src/seo_widgets.dart:195-219`

```dart
class _SeoSection extends StatelessWidget {
  final SectionRole role;               // хранится, никогда не используется
  Widget build(...) => Column(children: children);   // нет <section>/<main>/<nav>
}
class _SeoList extends StatelessWidget {
  final bool ordered;                   // хранится, никогда не используется
  Widget build(...) => Column(children: items);       // нет <ol>/<ul>/<li>
}
```

Ядро-модель полностью поддерживает `SectionNode` (выдаёт `<section>/<article>/<nav>/
<header>/<footer>/<main>`) и `ListNode` (выдаёт `<ol>/<ul>`) - см.
`html_serializer.dart:103-116`. Но surface Flutter-виджетов намеренно их
уплощает: `Seo.section(role: SectionRole.main)` выдаёт **никакого `<main>`**, а
`Seo.list(ordered: true)` выдаёт **никакого `<ol>`**. Параметры `role` и `ordered`
мертвы для своей очевидной SEO-цели.

Доказательство, что это намеренно и зашипено: тест ассертит уплощение, а не ловит
его (`test/seo_widgets_test.dart:148-150`):

```dart
final body = seal().body;
expect(body, hasLength(1));
expect(body[0], isA<ParagraphNode>());   // НЕ SectionNode
```

Почему это важно: семантическое секционирование (`<main>`, `<nav>`, `<ol>`) - это
базовый SEO/accessibility-сигнал, ровно то ценностное предложение этой библиотеки.
Пользователь, пишущий `Seo.section(role: main)`, разумно ожидает `<main>` в
`view-source` и не получает его. Подробная документация
(`docs/flutter-widgets.md:93,99`) тихо признаёт «role задаёт визуальную
группировку» / «только визуальная подсказка», но README пакета всё равно
перечисляет их под «регистрирует семантическую информацию», а сигнатура API
сильно намекает на обратное. Либо протяните `role`/`ordered` до
`SectionNode`/`ListNode`, либо уберите параметры, чтобы API перестал обещать то,
чего не выполняет.

### 3.2 MAJOR - README-«автоматический bot-aware транспорт» молча отключается при сконфигурированном кэше

`packages/hydraline_server/lib/src/middleware.dart:102-131`

Когда `config.cache != null`, каждый ответ идёт через `_cachedResponse`, который
всегда возвращает буферизованный `200` с `Content-Length`. Ветка bot-детекции
(буферизация-для-ботов vs chunked-для-людей) на строках 128-131 достижима только
когда `cache == null`. То есть две флагманские фичи - кэширование и bot-aware
стриминг-транспорт - взаимоисключающи, чего не указывает ни один документ. Это не
баг корректности (тела всё ещё байт-идентичны), но реальный разрыв между
разрекламированным и реализованным поведением.

### 3.3 MAJOR - Имя кэша service worker никогда не ротируется; устаревший движок раздаётся бесконечно

`packages/hydraline_flutter/lib/src/assets/js_service_worker.dart:20,25-56`

```js
var CACHE_NAME = 'hydraline-v1';        // захардкожено, никогда не бампится
var ENGINE_ASSETS = ['main.dart.js', 'canvaskit'];
// install -> skipWaiting(); activate -> clients.claim(); fetch -> cache-first
```

Cache-first на `main.dart.js` (который Flutter по умолчанию не хэширует по
содержимому) с фиксированным именем кэша и без инвалидации версии означает, что
возвращающимся посетителям может раздаваться устаревший Flutter-движок после
редеплоя, без встроенного механизма сброса. Для библиотеки, которая шипит service
worker как фичу, это требует стратегии версий/хэшей или явного предупреждения «вы
сами отвечаете за cache-busting».

### 3.4 MAJOR - Мёртвый метод интерфейса `RouteAdapter.navigateToForExtraction`

`packages/hydraline_flutter/lib/src/route_adapter.dart:20,47,61`

Обе реализации пусты (`async {}`); SSG-раннер никогда его не вызывает (он
manifest-driven и лишь держит `_adapter` как «reserved for widget-based
extraction», `ssg_runner.dart`). Метод задокументирован в
`docs/flutter-widgets.md:365` как часть публичного интерфейса. Это мёртвая
поверхность, подразумевающая возможность (навигация по дереву виджетов для
извлечения), которой не существует.

### 3.5 MINOR - Vanilla-island JS падает на отсутствующих под-элементах (прерывает гидратацию)

`packages/hydraline/lib/src/assets/vanilla_islands.dart:36-37,40-43`

```js
root.querySelector('[data-carousel-prev]').addEventListener(...); // null-deref если отсутствует
...
function CopyButton(root){var btn=$('[data-copy-target]',root)||root.querySelector('button');
var targetId=attr(btn,'data-copy-target'); ...   // attr(null,...) падает до гарда
```

`Carousel` и `CopyButton` разыменовывают результаты `querySelector` без
null-гарда (в отличие от `Accordion`, `LazyImage`, `Theme`, которые проверяют).
Если у карусели нет кнопок prev/next, или у copy-button нет ни `[data-copy-target]`,
ни `<button>`, гидратация бросает `TypeError`. У цикла бутстрапа
(`vanilla_islands.dart:62-65`) нет `try/catch`, поэтому один сломанный остров может
прервать гидратацию остальных островов на странице.

### 3.6 MINOR - `SsgCollector` молча теряет данные (записи после seal, перезапись meta)

`packages/hydraline/lib/src/collector.dart:40-43,82-87`

```dart
void _add(DocumentNode node, String? key) {
  if (_sealed != null) return;   // молча отбрасывает
  ...
}
void addMeta(SeoMeta meta) {
  if (_sealed != null) return;   // молча отбрасывает
  _meta = meta;                  // молча перезаписывает предыдущий addMeta
}
```

Дедупликация-по-ключу намеренна и норм, но молчаливый сброс каждой записи после
`seal()` и молчаливая перезапись второго `addMeta` - это footgun'ы, превращающие
ошибки программиста в трудноотлаживаемые пустые/некорректные страницы. Лучше
`StateError` на мутацию после seal и на двойной `addMeta`.

### 3.7 MINOR - HEAD-запросы обрабатываются как GET (полное тело строится и возвращается)

`packages/hydraline_server/lib/src/middleware.dart:75-134`

Хендлер никогда не инспектирует `request.method`. `HEAD` строит документ,
сериализует его, (опционально) кэширует и возвращает `Response` с прикреплённым
телом. `shelf_io` может отрезать тело на проводе, но сама middleware делает лишнюю
работу и не соблюдает семантику HEAD на уровне хендлера.

### 3.8 MINOR - Параметр `data` у `DocumentBuilder` всегда `null`

`packages/hydraline_server/lib/src/middleware.dart:144`

```dart
return builder(request, null);
```

Тайпдеф - `Function(Request, Object? data)`, но ничего никогда не передаётся.
Метаданные сматченного маршрута и значения динамических сегментов не пробрасываются;
билдеры вынуждены сами перепарсивать `request.url.pathSegments` (как делает пример).
Второй параметр сейчас - чистая церемония.

### 3.9 MINOR - In-memory кэш ограничивает число записей, но не их размер

`packages/hydraline_server/lib/src/cache.dart:19-46`

`maxSize` ограничивает число записей (по умолчанию 500), но не байты на запись. 500
больших страниц - это на практике неограниченная память. Для зашипленного кэша
уместен бюджет по байтам (или хотя бы задокументированная оговорка). Также заметьте,
что мутации map безопасны только в single-isolate модели shelf (сегодня норм, но
стоит комментария до того, как кто-то расшарит его между изолятами).

### 3.10 MINOR - `_redirectResponse` обрабатывает 303 несогласованно с `Http.redirect`

`packages/hydraline_server/lib/src/middleware.dart:147-152` vs
`http_semantics.dart:27-34`

У `Http.redirect` есть явная ветка `303 => Response.seeOther(...)`;
`_redirectResponse` (используется, когда билдер бросает
`RedirectException(status:303)`) падает в дженерик `Response(303,
headers:{location})`. Два пути кода редиректа, два поведения. Функционально близко,
но несостыковка реальна.

### 3.11 MINOR - `SafeUrl.tryParse('')` успешен, давая `href=""` / `src=""`

`packages/hydraline/lib/src/escaping.dart:71-78`

У пустой или состоящей из пробелов строки нет схемы, поэтому она обходит
отклонение по allowlist и возвращает валидный `SafeUrl('')`. В HTML
`href=""`/`src=""` - это самоссылка (перезапрашивает текущую страницу). Не XSS, но
footgun, который тип в остальном спроектирован предотвращать.

### 3.12 MINOR - Sitemap `lastmod` использует поля локального времени, а не UTC

`packages/hydraline/lib/src/sitemap.dart:150-155`

`_date` читает `dt.year/month/day` напрямую. `DateTime.now()` (локальный) около
полуночи в таймзоне позади UTC может выдать дату, смещённую на день. Конвертируйте
в `.toUtc()` или задокументируйте, что вызывающие обязаны.

### 3.13 MINOR - `FutureBuilder` в `IslandHost` игнорирует ошибки

`packages/hydraline_flutter/lib/src/island_host.dart:70-73`

```dart
return FutureBuilder<Widget>(
  future: factory(binding.state),
  builder: (context, snapshot) => snapshot.data ?? const SizedBox.expand(),
);
```

`snapshot.hasError` / `connectionState` игнорируются, поэтому падающая фабрика
острова рендерится как пустой бокс без диагностики. Дополнительно: *синхронный*
throw внутри `factory(...)` полностью обходит `FutureBuilder` (вызов делается жадно
на строке 71). Провалившийся отложенный `loadLibrary()` проявится как пустой регион
в продакшене без сигнала.

### 3.14 NIT - Расхождение именования: виджет `props` vs узел/спека `state`

`island.dart` экспонирует `props`; `island_manifest.dart` и `document_node.dart`
используют `state`; выдаваемый атрибут - `data-state`. Одна концепция, три имени.

### 3.15 NIT - `Seo.text(headingLevel:)` дублирует `Seo.heading(level:)`

`seo_widgets.dart:17-28` - две публичные точки входа выдают один и тот же
`HeadingNode`. Небольшая избыточность API.

### 3.16 NIT - Регистр имён заголовков ответа несогласован

`middleware.dart:171-177` выдаёт title-case `ETag`/`Vary`/`Cache-Control`, тогда как
`x-robots-tag` (строки 162 и `http_semantics.dart:60`) в нижнем регистре. Безвредно
в HTTP/1.1 и нормализуется в HTTP/2, но несогласованно в пределах одного файла.

### 3.17 NIT - Пробелы в защите-в-глубину при интерполяции сырых строк

- `assets.dart:75-79` интерполирует `baseHref` в инлайновый `<script src="...">`
  через `UnsafeHtmlNode` без экранирования. Источник - конфиг, не пользовательский
  ввод, поэтому риск низкий, но это единственное место, где кодовая база выходит за
  рамки собственной дисциплины экранирования.
- `htmx.dart:35-41,91` пишет строки вызывающего в заголовки `HX-*` без валидации
  CRLF. На практике слой заголовков `dart:io` отклоняет CRLF, поэтому это в основном
  смягчается на транспорте; всё же стоит валидирующего конструктора для
  защиты-в-глубину.
- `js_custom_element.dart:65-67` конкатенирует части `data-size` в CSS-строку без
  валидации на стороне JS (сегодня безопасно, потому что Dart-сериализатор всегда
  выдаёт только целые числа, и это заключено в Shadow DOM).

### 3.18 NIT - `bin/build.dart` обходит собственную публичную поверхность пакета

`packages/hydraline_flutter/bin/build.dart:3-4` импортирует
`package:hydraline_flutter/src/...` напрямую вместо курируемого
`package:hydraline_flutter/build.dart`, нарушая инкапсуляцию, которую рекламирует
пакет.

## 4. Ложные тревоги (заявления, которые НЕ подтвердились при проверке)

Они были помечены в ходе ревью, но **не** являются реальными дефектами -
задокументированы здесь, чтобы их не «пофиксили» в регрессии.

- **«unescape в IslandStateCodec - не корректная инверсия» - ЛОЖЬ.** Round-trip
  encode/decode протестирован напрямую на входах, содержащих `&quot;`,
  `&amp;quot;`, `&#39;`, `<`, `>`, `"`; все корректно проходят round-trip (encode
  экранирует JSON-строку целиком, decode обращает это). Никакой порчи.
- **«Стрип префикса `assets/hydraline/` - мёртвый код / возвращает 404» - ЛОЖЬ.** В
  shelf `request.url.path` - это *относительный* URL без ведущего слэша, поэтому
  `path.startsWith('assets/hydraline/')` матчится как задумано. Подтверждено
  проходящим тестом `robots_assets_test.dart` («serves assets under the
  assets/hydraline/ prefix too»).
- **«Сравнения путей в example/main.dart (`'api/faq'`, `'robots.txt'`) никогда не
  матчатся» - ЛОЖЬ.** Та же семантика относительного URL shelf; сравнения
  корректны.
- **«`'/${request.url.path}'` создаёт двойной слэш `//`» - ЛОЖЬ.** По той же
  причине - у `request.url.path` нет ведущего слэша, поэтому это корректно создаёт
  единственный ведущий слэш.

Повторяемость непонимания относительного URL shelf - хорошее напоминание:
`Request.url` относителен к точке монтирования хендлера и никогда не несёт ведущий
слэш.

## 5. Заметки о согласованности, документации и тулинге

- **Docs vs код: в основном согласованы.** `docs/document-model.md`,
  `architecture.md` и `server.md` точно описывают набор узлов и сериализатор.
  Заметное расхождение - семантический пробел секционирования (3.1): документация
  Flutter-виджетов тихо понижает `role` до «только визуально», тогда как README
  пакета и чистая Dart-документация представляют секционирование как
  первоклассную SEO-фичу. Согласуйте это.
- **Версионирование согласовано:** все четыре пакета и пример закреплены на
  `0.0.3`; CHANGELOG'и попакетны и связны.
- **CI/гейты сильные:** `analyze` запускается с `--fatal-infos --fatal-warnings`,
  границы машинно-проверяемы, а `precommit` даже гоняет end-to-end SSG-аудит
  примера. Это лучшая строгость, чем у большинства репозиториев до 1.0.
- **Форма тестирования:** отличная широта на чистой Dart-стороне (сериализатор,
  экранирование, фаззинг, sitemap/robots, route/island манифесты,

  JS-рантайма и L2-островов, ровно та область с наибольшим новаторским риском.
  Репозиторий сам отмечает отсутствие Playwright. Добавление даже минимального
  headless-браузерного smoke-теста гидратации существенно поднимет уверенность.
- **`analysis_options.yaml`, `.gitattributes` (golden binary), игнор TEMP,
  English-only конвенция:** всё присутствует и соблюдается.

## 6. Приоритизированные рекомендации

Обязательно до заявления о «продакшене» / 1.0:
1. Исправьте или уберите семантический пробел `Seo.section`/`Seo.list` (3.1). Это
   напрямую подрывает ключевое SEO-обещание на Flutter-surface.
2. Добавьте реальный браузерный smoke-тест гидратации L2-островов + JS-диспетчера +
   service worker. Самый сложный, наименее проверенный код не должен шипаться на
   одних лишь ассертах Dart-строк.
3. Задокументируйте (или снимите) взаимоисключаемость кэша и bot-aware транспорта
   (3.2) и историю cache-busting service worker (3.3).

Стоило бы:
4. Null-гарды для vanilla `Carousel`/`CopyButton` и обёртка цикла бутстрапа
   гидратации в `try/catch`, чтобы один плохой остров не мог прервать страницу
   (3.5).
5. Заставьте `SsgCollector` громко падать при записях после seal / двойном meta
   (3.6).
6. Всплывайте ошибки фабрики острова в `IslandHost` (3.13).
7. Либо используйте параметр `data` у `DocumentBuilder`, либо уберите его (3.8);
   добавьте байтовый бюджет кэша (3.9).

Приятно иметь:
8. Согласуйте именование (`props`/`state`), регистр заголовков и перекрытие
   `Seo.text` с heading; уберите мёртвый `RouteAdapter.navigateToForExtraction`
   (3.4) или реализуйте его; пустой `SafeUrl` и UTC-даты sitemap (3.11, 3.12).

## 7. Вердикт

Плотный, хорошо протестированный, безопасность-осознающий кодовый базис,
адресующий реальную и актуальную проблему, с чистой архитектурой и
зафиксированными границами. Чистое Dart-ядро (`hydraline`, `hydraline_server`)
заслуживает доверия и близко к продакшену. Слой Flutter-островов
`hydraline_flutter` - амбициозная, отличающая часть, - и одновременно наименее
проверенная; он требует валидации в реальном браузере и одного настоящего
семантического исправления, прежде чем верхнеуровневое заявление
«production-ready Flutter Web SEO» будет полностью заслужено. Критических багов
или эксплуатируемых дыр безопасности не найдено; проблемы - это смесь одной
значимой отсутствующей-фичи (3.1), нескольких разрывов «реклама vs факт» и
обычного хвоста краевых случаев устойчивости.

**Оценка: B+ (крепкая до-1.0). Готово к использованию L0/L1 сегодня;
пилотируйте L2 с браузерным тестированием.**
