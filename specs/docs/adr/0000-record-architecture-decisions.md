# ADR-0000: Вести журнал архитектурных решений (ADR)

- **Статус:** Accepted
- **Дата:** 2026-07-13
- **Контекст фазы:** Phase 0 (cross-cutting)
- **Связано:** `DEVELOPMENT.md §11 (DoD-1)`, `CONTRIBUTING.md §6`

## Контекст

Проект Hydraline имеет обширную предысторию решений (спека v3.6, ADR_OPEN_QUESTIONS
Q1–Q10, RISK_ANALYSIS). По мере реализации будут возникать новые нетривиальные
решения. Нужен лёгкий, версионируемый механизм фиксации «почему», чтобы решения
были воспроизводимы, а не «утеряны» в PR-обсуждениях.

## Рассмотренные варианты

1. **ADR-файлы в репозитории (`docs/adr/`)** — markdown, один файл на решение,
   нумерация, статусы. Diff-friendly, живут рядом с кодом.
2. **Только PR-описания** — решения тонут в истории, плохо ищутся.
3. **Внешняя wiki** — рассинхрон с кодом, отдельный источник правды.

## Решение

Принят вариант 1. ADR ведутся в `docs/adr/NNNN-<slug>.md` по `docs/adr/TEMPLATE.md`.
Существующие решения продукта остаются в `ADR_OPEN_QUESTIONS.md` (Q1–Q10) и спеке;
новые/изменяющие решения оформляются как ADR. Изменение решения — новый ADR со
ссылкой «Supersedes ADR-XXXX».

## Последствия

- **Плюсы:** воспроизводимость решений; лёгкий ревью; единый стиль; связь с инвариантами.
- **Минусы:** небольшой процессный оверхед (компенсируется шаблоном).
- **Влияние:** DoD-1 (`DEVELOPMENT.md §11`) требует ADR для нетривиальных решений.

## Проверка

Наличие ADR для нетривиальных решений проверяется на ревью PR (чек-лист
`CONTRIBUTING.md §4/§6`).
