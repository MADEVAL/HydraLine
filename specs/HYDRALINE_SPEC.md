## 1. Проблема

### 1.1 Суть

Flutter Web по умолчанию рендерит UI в `<canvas>` через CanvasKit/Skwasm. Первый HTTP-ответ
(`index.html`) — это пустой **app-shell**: скрипты загрузки движка и почти полное отсутствие
семантического контента. Реальный контент появляется в DOM **только после** скачивания и
инициализации движка и исполнения Dart-кода.

### 1.2 Кто ломается

- **Соц-краулеры и превью-боты** (Facebook, X/Twitter, LinkedIn, Telegram, WhatsApp, iMessage,
  Slack, Discord) — **не исполняют JavaScript**. Они забирают HTML, читают `og:*` / `twitter:*`
  и уходят. Видят пустой shell → превью не строится.
- **Простые парсеры / no-JS среды / аналитические краулеры** — то же самое.
- **Поисковые боты** — Googlebot умеет исполнять JS, но с задержкой и бюджетом; контент,
  доступный только после тяжёлой инициализации движка, индексируется хуже и медленнее.

### 1.3 Почему существующие пакеты не решают проблему

Все текущие пакеты делают **runtime DOM-инъекцию** — мутируют `document.head`/DOM уже
**после** загрузки Flutter:

| Пакет / подход | Что делает | Почему не закрывает дыру |
|---|---|---|
| `meta_seo` | мутирует meta-теги в рантайме | тегов **нет** в `view-source`; соц-боты их не видят (подтверждено на StackOverflow) |
| `seo` / `seo_renderer` | вставляет скрытый DOM в рантайме | контент появляется только после старта движка; для no-JS ботов пусто |
| `webify_toolkit` | набор runtime-хелперов | та же природа: post-boot мутация |
| Firebase Functions + user-agent sniffing | Cloud Function отдаёт HTML только ботам | это **cloaking** (разный контент боту и пользователю) → риск санкций поисковиков; хрупко; привязано к Firebase; ручной HTML-стринг |
| `--web-renderer html` | старый HTML-renderer | **удалён** из Flutter (issue #145954); не будущеустойчив; и всё равно не давал SEO |

### 1.4 Позиция команды Flutter (подтверждено)

- #171598 (Native SEO Support) — **закрыт как duplicate**.
- #181127 (RFC: Opt-in Semantic Shadow Tree) — **закрыт как out-of-scope** («улучшайте
  существующую semantics-систему»).
- #183992 (FlutterWebKitX) — **закрыт not_planned / r: invalid** («делайте как community package»).
- #187663 (route-level document rendering, `WebRouteRenderMode`) — **открыт**, вынесен в
  публичный design-doc, назначен на web-команду; это официально признанное **желаемое**
  направление, но **не поставленный продукт** и без сроков.
- #46789 (SEO indexability, 610 👍) — годами «P2 / скоро не будет».

**Вывод:** ядро Flutter в обозримом будущем не отдаёт настоящий семантический HTML в первом
ответе. Официальный дизайн (#187663) описывает **ровно то, что нужно**, но реализации нет.
Дыра свободна и её должен закрыть community-пакет, совместимый по модели с #187663.

### 1.5 Что именно отсутствует на рынке

Нет пакета, который **отдаёт настоящий семантический HTML в первом HTTP-ответе**
(build-time или per-request), а Flutter затем **гидрирует поверх** интерактивные зоны,
**не требуя переписывания приложения и не будучи новым фреймворком**.

Ближайший аналог — **Jaspr** (Dart SSR-фреймворк с islands/hydration). Но Jaspr — это
**отдельный фреймворк**: контент пишется его компонентами (не Flutter-виджетами), а не
интегрируется в существующее Flutter-приложение. Hydraline занимает свободную нишу:
**«оставь своё Flutter-приложение как есть, добавь SEO-документные и гибридные маршруты»**.

## 2. Цели, не-цели, принципы

### 2.1 Цели (Goals)

1. **G1.** Отдавать валидный, семантический HTML (заголовки, параграфы, ссылки, alt,
   метаданные, JSON-LD) в **первом HTTP-ответе** для маршрутов, помеченных как `document`/`hybrid`
   — доступный в `view-source`, без исполнения JS.
2. **G2.** Позволять Flutter **гидрировать** интерактивные зоны поверх этого HTML (**islands**),
   не перерисовывая статический контент.
3. **G3.** **Не требовать переписывания** существующих приложений: `app`-маршруты работают
   как сейчас; SEO-режимы включаются **пер-маршрут**, аддитивно.
4. **G4.** Дать полный SEO-инструментарий: метаданные, canonical, OG/Twitter, JSON-LD,
   `sitemap.xml`, `robots.txt`, статус-коды, редиректы, `noindex`, hreflang.
5. **G5.** Поддержать **оба пути доставки HTML**: build-time SSG (для CDN/Firebase/статик-хостинга)
   и request-time SSR (shelf / Dart Frog).
6. **G6.** Дать инструмент верификации: «что реально видит краулер» (devtools-оверлей + CLI-аудит).
7. **G7.** **Никакой cloaking**: боту и пользователю отдаётся **один и тот же** контент.

### 2.2 Не-цели (Non-Goals) — жёсткие границы

1. **NG1.** Hydraline **не фреймворк**. Он не владеет `main()`, не диктует роутер, не заменяет
   `MaterialApp`. Это библиотеки, встраиваемые в существующий проект.
2. **NG2.** **Не** автоматическая конвертация произвольных Flutter-виджетов в HTML/CSS.
   Семантика **явная** (developer-controlled), как и предлагает #187663 и #183992.
3. **NG3.** **Не** возрождение HTML-renderer; **не** замена CanvasKit/Skwasm.
4. **NG4.** **Никаких** C/C++/Rust/FFI/WASM-прослоек собственной разработки. Только Dart.
5. **NG5.** **Не** требует конкретного бэкенда/хостинга (Firebase — лишь один из адаптеров).
6. **NG6.** **Не** пытается сделать пиксель-в-пиксель одинаковую картинку в HTML и в CanvasKit.
   `document`-контент — это семантический документ, а не рендер виджетов.
7. **NG7.** **Не** ORM/CMS/data-layer. Данные предоставляет разработчик.

### 2.3 Инженерные принципы

- **P1. Pure-Dart ядро.** Слой моделей и сериализации не зависит от Flutter → тестируется
  `dart test` и переиспользуется на сервере.
- **P2. Одна модель — два источника.** И серверный pure-Dart билдер, и Flutter-виджеты
  сходятся к единому `DocumentNode`-дереву (см. §5.4).
- **P3. Безопасность по умолчанию.** Любой пользовательский текст экранируется/санитизируется
  на выходе в HTML; небезопасный HTML — только через явный, задокументированный opt-in.
- **P4. Прогрессивное улучшение.** Контент работает без JS; интерактив добавляется островами.
- **P5. Совместимость по контракту с #187663.** Термины и модель (`WebRouteRenderMode`,
  `DocumentNode`, island-манифест) осознанно совпадают, чтобы при появлении нативной поддержки
  миграция была тривиальной.
- **P6. Минимум runtime-JS.** Единственный собственный JS — маленький island-загрузчик
  (см. §7); всё остальное — стандартный `flutter_bootstrap.js`.

## 3. Позиционирование и конкурентный анализ

| Решение | HTML в первом ответе | Гидрация Flutter поверх | Переписывание проекта | Фреймворк | Cloaking-риск |
|---|---|---|---|---|---|
| `meta_seo` / `seo` / `webify_toolkit` | ❌ (runtime) | n/a | нет | нет | нет |
| Firebase Function + UA-sniff | ⚠️ только ботам | ❌ | частично | нет | **да** |
| Jaspr | ✅ | ❌ (свои компоненты, не Flutter islands) | **да (на Jaspr)** | **да** | нет |
| Нативный #187663 (будущее) | ✅ | ✅ | нет | нет (ядро) | нет |
| **Hydraline** | ✅ | ✅ (multi-view islands) | **нет** | **нет** | **нет** |

**Ключевой дифференциатор:** Hydraline — единственный путь получить *настоящий HTML в первом
ответе* **и** *Flutter-острова поверх*, **сохранив существующее Flutter-приложение** и **не
становясь фреймворком**.

## 4. Термины и режимы рендера

### 4.1 Режимы маршрута (`WebRouteRenderMode` — по #187663)

| Режим | Владелец контента | Типичное применение |
|---|---|---|
| `app` | CanvasKit/Skwasm (как сейчас) | дашборды, редакторы, приватные экраны |
| `document` | семантический HTML | блог, docs, лендинги, карточки товаров, справка |
| `hybrid` | HTML владеет SEO-контентом, Flutter-острова — интерактивом | товар с калькулятором/конфигуратором, статья с виджетом, карта |

Режим задаётся **пер-маршрут**. В одном приложении сосуществуют все три.

### 4.2 Основные термины

- **Prerender** — генерация HTML до/вместо исполнения Flutter (build-time SSG или request-time SSR).
- **Hydration (гидрация)** — «оживление» уже отрисованного HTML: монтирование Flutter-острова
  в конкретный DOM-элемент через multi-view API, без перерисовки статики.
- **Island (остров)** — явно очерченная интерактивная зона, в которую монтируется Flutter-view.
- **`DocumentNode`** — узел абстрактного дерева документа (заголовок, параграф, ссылка,
  изображение, список, island-плейсхолдер, raw-fragment и т.д.); pure-Dart, не зависит от `dart:ui`.
- **Island-манифест** — сериализованное описание островов маршрута (id, точка монтирования,
  props, директива гидрации), встраиваемое в HTML и потребляемое island-загрузчиком.
- **Route manifest** — карта маршрутов → режим, метаданные, статус, источник контента.

## 5. Архитектура

### 5.1 Состав (monorepo, melos ≥ 7 на pub workspaces, Dart SDK ≥ 3.6)

| Пакет | Роль | Зависит от Flutter? |
|---|---|---|
| `hydraline` | **pure-Dart ядро**: `DocumentNode`-дерево, HTML-сериализатор, модель метаданных, sitemap/robots, island-манифест, escaping/sanitization | **Нет** |
| `hydraline_widgets` | **Flutter-виджеты**: `Seo.text/image/link/head/…`, `Island(...)`, `HydraApp`; двойная природа (визуал + описатель документа); клиентский island-загрузчик | Да |
| `hydraline_server` | **shelf / Dart Frog middleware**: рендер `DocumentNode`→HTML на запрос, встраивание island-манифеста, статусы/редиректы/`noindex`, кэш | **Нет** (server Dart) |
| `hydraline_build` | **build-time SSG**: обход маршрутов, извлечение `DocumentNode` из Flutter-дерева в headless-биндинге, запись статичных `.html` + `sitemap.xml` + манифестов | Да (build/test-биндинг) |
| `hydraline_devtools` | **оверлей-инспектор + CLI-аудит**: «что видит краулер», валидация метаданных, диагностика гидрации, проверка no-JS | Да (оверлей) / pure-Dart (CLI) |

Дополнительно (внутренний):
- `hydraline_lint` (**МОЖЕТ**, поздние фазы) — правила анализатора (напр. «island без размеров»,
  «пользовательский текст без escape»).

### 5.2 Диаграмма зависимостей

```
                 ┌───────────────────────────────┐
                 │        hydraline (core)         │  pure Dart, dart:ui-free
                 │  DocumentNode • serializer •    │
                 │  metadata • sitemap/robots •    │
                 │  island manifest • escaping     │
                 └───────────────────────────────┘
                    ▲          ▲            ▲
        ┌───────────┘          │            └───────────────┐
        │                      │                            │
┌───────────────┐   ┌────────────────────┐        ┌──────────────────┐
│ hydraline_    │   │  hydraline_server   │        │  hydraline_build  │
│ widgets       │   │  (shelf/Dart Frog)  │        │  (SSG, headless)  │
│ (Flutter)     │   │  pure server Dart   │        │  (Flutter build)  │
└───────────────┘   └────────────────────┘        └──────────────────┘
        ▲                                                   ▲
        └──────────────────┬────────────────────────────────┘
                           │
                   ┌───────────────────┐
                   │ hydraline_devtools │  оверлей + CLI-аудит
                   └───────────────────┘
```

**Правило зависимостей (ДОЛЖНО):** `hydraline` (core) **не импортирует** `flutter`/`dart:ui`.
`hydraline_server` **не импортирует** `flutter`. Нарушение — блокирующая ошибка CI.

### 5.3 Ключевое архитектурное ограничение (реализм)

**Flutter-виджеты нельзя исполнять на «голом» Dart-сервере** — фреймворк требует `dart:ui`,
которого нет в обычной серверной среде Dart VM. Следствия, зашитые в архитектуру:

1. **`hydraline_server` НИКОГДА не запускает Flutter-виджеты.** Он работает только с:
   - (a) **предсобранным route-манифестом** из `hydraline_build` (готовые HTML-фрагменты /
     `DocumentNode`-деревья), и/или
   - (b) **pure-Dart `DocumentNode`-билдерами**, которые разработчик пишет для динамических
     маршрутов (на вход — данные из БД/API, на выход — `DocumentNode`).
2. **Извлечение `DocumentNode` из Flutter-дерева происходит в среде, где `dart:ui` доступен:**
   build-time через **headless Flutter-биндинг** (та же природа, что у `flutter test` /
   `flutter_tester` — рендер дерева без GPU). Это делает `hydraline_build`.
3. **Динамический per-request SSR** контента, требующего актуальных данных, идёт через
   pure-Dart билдеры (путь b). Flutter-виджеты для таких страниц используются как **визуальный
   слой в рантайме приложения** и (опционально) как build-time шаблон, но **не** исполняются на сервере.

Это разделение — не компромисс, а корректная модель. Оно прямо отражает разграничение из
#187663 (build-time document rendering + server adapters + island bootstrap) и делает проект
**реализуемым без нативных прослоек**.

### 5.4 Две поверхности авторинга — одна модель (P2)

```
   Автор контента выбирает поверхность:

   (A) Flutter-виджеты           (B) Pure-Dart Document builder
       Seo.text(...)                 Document.of(context)
       Seo.image(...)                  ..h1(...)..p(...)..link(...)
       Island(...)                     ..island(id, props, directive)
          │                                  │
          │  build-time extraction           │  напрямую (сервер, per-request)
          │  (headless binding)              │
          ▼                                  ▼
        ┌──────────────────────────────────────────┐
        │            DocumentNode tree               │  ← единая модель (core)
        └──────────────────────────────────────────┘
                          │
                          ▼
                 HTML serializer (core)
                          │
             ┌────────────┴────────────┐
             ▼                          ▼
     статичные .html (SSG)     HTTP-ответ (SSR)
```

- Поверхность **(A)** — удобна тем, кто хочет держать контент рядом с Flutter-кодом и
  переиспользовать визуал. Извлечение — только там, где есть `dart:ui` (build/тест-биндинг).
- Поверхность **(B)** — для динамики на сервере, где Flutter недоступен.
- Обе дают **идентичный** `DocumentNode`, поэтому один сериализатор, один island-механизм,
  один набор тестов на выход.

### 5.5 Поток данных: три сценария доставки

**Сценарий SSG (статик-хостинг / CDN / Firebase Hosting):**
```
flutter + hydraline_build  →  обход route-манифеста  →  headless-извлечение DocumentNode
   →  serializer  →  /dist/**.html + sitemap.xml + robots.txt + island-манифесты
   →  деплой на статик-хостинг  →  краулер видит готовый HTML; браузер грузит Flutter и гидрирует острова
```

**Сценарий SSR (динамика, shelf / Dart Frog):**
```
HTTP-запрос  →  hydraline_server middleware  →  match route  →
   [document/hybrid] pure-Dart DocumentNode builder(данные)  →  serializer  →  HTML + island-манифест
   [app]           →  отдать app-shell как обычно
   →  ответ со статусом/редиректом/noindex по правилам маршрута
```

**Сценарий гибрид (SSG для статики + SSR-фрагменты для динамики):** поддерживается; выбор —
на стороне интеграции. Server islands (по мотивам Astro `server:defer`) — **МОЖЕТ**, поздние фазы.

## 6. Функциональные требования по пакетам

### 6.1 `hydraline` (core, pure-Dart)

- **C-1 (ДОЛЖНО).** Модель `DocumentNode`: минимум — document, head/meta, heading (h1–h6),
  paragraph, text/span, link (`<a href>`), image (`<img src alt>`), list (ul/ol/li), section/article/nav/header/footer,
   time, blockquote, code/pre, table (базово), island-placeholder, raw-html-fragment (opt-in: `UnsafeHtmlNode`).
- **C-2 (ДОЛЖНО).** Модель метаданных: title, description, canonical, robots-директивы,
  Open Graph (`og:*`), Twitter Card (`twitter:*`), произвольные `<meta>`/`<link>`, charset/viewport,
  `lang`, hreflang-альтернативы.
- **C-3 (ДОЛЖНО).** JSON-LD: типобезопасное представление structured data (Article, Product,
  BreadcrumbList, WebPage, Organization и произвольные схемы) → корректный `<script type="application/ld+json">`.
- **C-4 (ДОЛЖНО).** HTML-сериализатор: детерминированный, стабильный порядок атрибутов,
  корректное экранирование, конфигурируемый pretty/minified вывод. **Golden-тесты** на выход.
- **C-5 (ДОЛЖНО).** Безопасность: escaping текста и значений атрибутов по контексту (HTML text,
  attribute, URL); валидация/санитайз URL (блок `javascript:` и пр.); raw-HTML — только через
   явный `UnsafeHtmlNode`-узел с предупреждением в API.
- **C-6 (ДОЛЖНО).** `sitemap.xml` (urlset, lastmod, changefreq, priority, alternates/hreflang)
  и `robots.txt` (правила, ссылка на sitemap) — генераторы.
- **C-7 (ДОЛЖНО).** Island-манифест: сериализация/десериализация (id, mount-selector/placeholder-id,
  props как JSON-safe, директива гидрации, размеры-заглушки для anti-CLS).
- **C-8 (ДОЛЖНО).** Route-манифест: маршрут → режим, метаданные, статус, источник контента, canonical.
- **C-9 (СЛЕДУЕТ).** Валидаторы: длина title/description, обязательность alt, дубли canonical,
  битые hreflang — как переиспользуемые проверки (используются devtools и lint).
- **C-10 (ДОЛЖНО).** Ноль зависимостей от Flutter/`dart:ui`; работает на Dart VM (Windows/WSL/CI).

### 6.2 `hydraline_widgets` (Flutter)

- **W-1 (ДОЛЖНО).** `Seo.*`-виджеты (text, image, link, heading, section, list, head/meta) —
  **двойная природа**: (1) визуально рендерят обычный Flutter-виджет в рантайме приложения,
  (2) несут описатель, из которого извлекается `DocumentNode`.
- **W-2 (ДОЛЖНО).** `Island(id, props, directive, placeholder, size)` — маркер интерактивной зоны:
  в рантайме — обычный виджет; при извлечении — island-placeholder + запись в манифест.
- **W-3 (ДОЛЖНО).** `HydraApp` / интеграционная обёртка — регистрация режимов маршрутов и
  предоставление контекста извлечения; **не заменяет** `MaterialApp`/роутер, а оборачивает/дополняет.
- **W-4 (ДОЛЖНО).** Совместимость с роутерами: **first-class** адаптер для `go_router`;
  общий интерфейс `RouteAdapter` для остальных (`auto_route` — best-effort через интерфейс,
  `Navigator 2.0` — fallback-адаптер, `beamer` — не поддерживается). Hydraline **не**
  навязывает конкретный роутер и не зависит от него в pubspec.
- **W-5 (ДОЛЖНО).** Клиентский **island-загрузчик** (Dart→JS): по island-манифесту находит
  плейсхолдеры и монтирует Flutter-view через multi-view API (`addView({hostElement, initialData})`),
  соблюдая директиву гидрации (см. §7).
- **W-6 (ДОЛЖНО).** Мульти-вью корневой виджет (`runWidget` + `ViewCollection` + `View`),
  сопоставляющий каждый `FlutterView` острову.
- **W-7 (СЛЕДУЕТ).** Хелперы anti-CLS: резервирование размеров острова до гидрации.

### 6.3 `hydraline_server` (shelf / Dart Frog, pure server Dart)

- **S-1 (ДОЛЖНО).** Middleware/handler для shelf **и** адаптер для Dart Frog.
- **S-2 (ДОЛЖНО).** Маршрутизация по route-манифесту: `document`/`hybrid` → рендер HTML;
  `app` → отдать app-shell.
- **S-3 (ДОЛЖНО).** Точка расширения «pure-Dart `DocumentNode` builder(request, data)» для
  динамических маршрутов (per-request SSR).
- **S-4 (ДОЛЖНО).** Встраивание island-манифеста и корректных абсолютных путей к ассетам
  Flutter (`/flutter_bootstrap.js`, `main.dart.js`, canvaskit) + `<base href>` при необходимости.
- **S-5 (ДОЛЖНО).** HTTP-семантика: статус-коды (200/301/302/404/410/5xx), редиректы,
  `X-Robots-Tag`/`noindex`, канонизация путей (path-routing, без hash).
- **S-6 (ДОЛЖНО).** Кэширование: заголовки `Cache-Control`/`ETag`, конфигурируемый TTL,
  in-memory/pluggable кэш отрендеренного HTML.
- **S-7 (СЛЕДУЕТ).** Стриминг ответа и «shell-first» отдача для больших страниц (поздние фазы).
- **S-8 (ДОЛЖНО).** Отдача `sitemap.xml` / `robots.txt` (сгенерированных ядром).
- **S-9 (НЕ ДОЛЖНО).** Никогда не исполнять Flutter-виджеты; не зависеть от `package:flutter`.

### 6.4 `hydraline_build` (SSG, build-time)

- **B-1 (ДОЛЖНО).** Обход route-манифеста и извлечение `DocumentNode` из Flutter-дерева в
  **headless-биндинге** (без GPU), детерминированно.
- **B-2 (ДОЛЖНО).** Запись `dist/`: `**.html` по маршрутам, `sitemap.xml`, `robots.txt`,
  island-манифесты, копирование/линковка Flutter web-ассетов.
- **B-3 (ДОЛЖНО).** Динамические сегменты: генерация страниц из предоставленного разработчиком
  набора параметров/данных (`/blog/:slug` × N).
- **B-4 (ДОЛЖНО).** Совместимость выхода со статик-хостингами (Firebase Hosting, Netlify,
  Cloudflare Pages, GitHub Pages): корректные rewrites/fallback-рекомендации в доке (без
  cloaking).
- **B-5 (СЛЕДУЕТ).** Инкрементальная сборка / кэш по хэшу входных данных.
- **B-6 (ДОЛЖНО).** CLI-интерфейс (`dart run hydraline_build ...`) с воспроизводимым выводом.

### 6.5 `hydraline_devtools` (инспектор + аудит)

- **D-1 (ДОЛЖНО).** CLI-аудит (pure-Dart): забрать `view-source`/no-JS-рендер маршрута,
  показать «что видит краулер» (title, meta, og, JSON-LD, заголовки, ссылки, alt), выдать
  отчёт и exit-code для CI.
- **D-2 (ДОЛЖНО).** Валидации SEO (через core-валидаторы C-9): длины, дубли, отсутствие alt,
  битые canonical/hreflang, отсутствующие og-поля.
- **D-3 (СЛЕДУЕТ).** Оверлей в dev-режиме приложения: подсветка островов, их директив,
  границ HTML-vs-Flutter, диагностика несостоявшейся гидрации.
- **D-4 (СЛЕДУЕТ).** Проверка соответствия «SSG-HTML ↔ гидрированный DOM» (регрессия рассинхрона).
- **D-5 (МОЖЕТ).** Симуляция user-agent’ов ботов (только для аудита; продукт cloaking не делает).

## 7. Острова и гидрация (детально)

### 7.1 Механизм (на подтверждённом multi-view API)

1. Prerender вставляет в HTML **плейсхолдер острова**, например семантический контейнер с
   атрибутами `data-hydraline-island`, `data-id`, `data-props`, `data-directive`, зарезервированными
   размерами и (опц.) fallback-контентом для no-JS.
2. Грузится стандартный `flutter_bootstrap.js`; движок инициализируется в режиме
   **`multiViewEnabled: true`** (одна инстанция движка на страницу — несколько views;
   несколько **инстанций** движка на страницу не поддерживаются, это учтено).
3. **Island-загрузчик** (из `hydraline_widgets`, §6.2 W-5) читает манифест, для каждого острова
   по его **директиве** ждёт триггер и затем монтирует Flutter-view в плейсхолдер:
   `flutterApp.addView({ hostElement, initialData: props })`. Dart-сторона рендерит остров
   через `ViewCollection`/`View`.

### 7.2 Директивы гидрации (модель по мотивам Astro, но нативно для Flutter)

| Директива | Триггер | Браузерное API | Применение |
|---|---|---|---|
| `hydrateOnLoad` | сразу после загрузки | `DOMContentLoaded` | критичный интерактив «выше сгиба» |
| `hydrateOnIdle` | простой | `requestIdleCallback` (+fallback) | второстепенное |
| `hydrateOnVisible` | попадание в вьюпорт | `IntersectionObserver` | ниже сгиба, тяжёлые виджеты |
| `hydrateOnMedia(q)` | совпадение media-query | `matchMedia` | только desktop/mobile |
| `hydrateManual` | по вызову API | — | по клику/явному триггеру |

- **Порядок:** гидрация **top-down** (родительский остров раньше вложенного).
- **Директива по умолчанию — `hydrateOnIdle`** (не `hydrateOnLoad`), чтобы не бить по TTI
  (см. R3 в `RISK_ANALYSIS.md`); `hydrateOnLoad` — только явный выбор разработчика.
- **Props:** передаются JSON-safe; сложные типы недопустимы через границу (документируется,
  подробности — ADR Q6).
- **Anti-CLS (ДОЛЖНО):** до гидрации плейсхолдер держит зарезервированные размеры; при
  `hydrateOnVisible` учитывать, что скрытые (`display:none`/`height:0`) контейнеры не триггерят
  observer — валидатор devtools это отлавливает.

### 7.3 Ограничения и риски островов (учтены, см. §14)

- **Регрессия multi-view sizing в Flutter 3.41.x** (issue #185034) — canvas острова может не
  совпадать с host-контейнером. → политика поддерживаемых версий Flutter + интеграционные
  тесты размеров + документированный workaround/пин.
- **Стоимость старта движка** (загрузка `main.dart.js`/canvaskit) → дефолт на отложенные
  директивы (`idle`/`visible`), «скелетоны» в HTML.
- **Единственная инстанция движка** → все острова делят один движок; общее состояние между
  островами — через явную шину/URL/`localStorage`, не через общий Flutter-контекст.

## 8. Модель документа, сериализация, безопасность

- **M-1.** `DocumentNode` — иммутабельное дерево; сборка через билдер (поверхность B) или
  извлечение (поверхность A). Узлы несут только семантику, не «пиксельный» стиль.
- **M-2.** Сериализатор — единственный источник HTML. Детерминизм (стабильный порядок,
  предсказуемый вывод) — обязательное свойство для golden-тестов.
- **M-3. Контекстное экранирование:** текст ≠ значение атрибута ≠ URL. Отдельные функции
  экранирования; URL-санитайз с allowlist схем (`http`, `https`, `mailto`, относительные).
- **M-4. UnsafeHtmlNode** только через явный узел; в API — предупреждающий комментарий и,
   желательно, требование прохождения санитайзера, предоставляемого разработчиком.
- **M-5.** Никакого cloaking на уровне модели: один и тот же `DocumentNode` → один и тот же
  HTML вне зависимости от user-agent.

- **SEO-1.** Пер-маршрутные `title`, `description`, `canonical`.
- **SEO-2.** Open Graph (`og:title/description/image/url/type/locale/site_name`, image:secure/alt/размеры).

- **SEO-11.** Семантика: настоящие `h1..h6`, `nav`, `article`, `main`, `a[href]`, `img[alt]`.

## 10. Нефункциональные требования

- **NF-1. Производительность SSR:** median рендер `document`-маршрута (без учёта данных)
  — целевой бюджет задаётся в фазе бенчмарков; сериализатор — без квадратичных алгоритмов.
- **NF-2. TTFB/кэш:** поддержка кэш-заголовков и повторного использования отрендеренного HTML.
- **NF-3. Core Web Vitals:** нулевой/минимальный CLS от островов (резерв размеров); JS-бюджет
  island-загрузчика — «маленький» (порог фиксируется в фазе perf).
- **NF-4. Совместимость Flutter:** явная матрица поддерживаемых версий; guard против известной
  регрессии multi-view (#185034); CI на нескольких Flutter-каналах.
- **NF-5. Безопасность:** отсутствие XSS в выводе (fuzzing/property-тесты escaping), отсутствие
  секретов в логах, санитайз URL.
- **NF-6. Доступность (a11y):** семантический HTML и alt по умолчанию улучшают доступность;
  валидаторы предупреждают об отсутствии alt/пустых ссылках.
- **NF-7. Кроссплатформенность разработки:** все pure-Dart пакеты собираются/тестируются на
  Windows **и** WSL; отсутствие платформозависимых путей/переносов строк в тестах (нормализация
  `\r\n`/`\n`).
- **NF-8. Наблюдаемость:** структурные логи сервера, коды выхода CLI-аудита для CI.
- **NF-9. Версионирование:** SemVer, автогенерация changelog через Conventional Commits (melos).
- **NF-10. Лицензия/публикация:** пригодность к публикации на pub.dev (описание, пример,
  документация, `>=` ограничения).

## 11. Режим разработки и стратегия тестирования

### 11.1 Общий цикл на каждую фазу

```
Brainstorm  →  Spike (одноразовый прототип для снятия неопределённости)  →
   заморозка контракта/API  →  TDD (red → green → refactor)  →
   полное покрытие + golden/property/integration  →  аудит devtools  →  ретро
```

- **Brainstorm.** Для каждой фазы — короткий design-note: варианты, риски, решение (ADR).
- **Spike.** Отдельно от продакшн-кода (в `spikes/`), только для проверки гипотез (напр.
  headless-извлечение `DocumentNode`, монтирование острова через `addView`). Результат spike —
  **не** мёржится как есть; переписывается по TDD.
- **TDD.** Ни одна строка продакшн-логики без предшествующего падающего теста.

### 11.2 Пирамида тестов

| Уровень | Где | Инструмент | Что покрывает |
|---|---|---|---|
| Unit | core, server, build | `dart test` | модель, сериализатор, escaping, sitemap/robots, манифесты, билдеры |
| Golden (HTML) | core | `dart test` + сравнение эталонов | детерминированный HTML-вывод (с нормализацией переносов строк) |
| Property/fuzz | core | `dart test` (генеративные) | escaping/санитайз (нет XSS на случайных входах) |
| Widget | widgets | `flutter test` | двойная природа `Seo.*`, извлечение `DocumentNode` в headless-биндинге |
| Integration (server) | server | `dart test` + `shelf`/Dart Frog test-harness | статусы, редиректы, noindex, кэш, встраивание манифеста |
| Build/SSG | build | `flutter test`/CLI e2e | корректность `dist/`, sitemap, динамические сегменты |
| E2E (браузер) | widgets/devtools | headless-браузер (Playwright/Puppeteer через CI) | реальная гидрация островов, no-JS-фоллбэк, CLS |
| Audit (краулер) | devtools | CLI | «что видит краулер», SEO-валидаторы, exit-code для CI |

### 11.3 Полное покрытие

- **Порог покрытия (ДОЛЖНО):** ≥ 90% строк/веток для `hydraline` (core) и `hydraline_server`;
  ≥ 80% для остальных (с обоснованными исключениями для тонкого JS-interop, покрываемого E2E).
- Покрытие измеряется в CI (`dart test --coverage` / `flutter test --coverage`), публикуется
  как артефакт; падение ниже порога — блокирующая ошибка.

### 11.4 Тестовая среда Windows / WSL

- **Матрица CI (ДОЛЖНО):** pure-Dart пакеты — Windows-runner **и** Linux(WSL-эквивалент);
  Flutter-пакеты — на обеих ОС в пределах поддерживаемых Flutter-версий.
- **Правила кроссплатформенности (ДОЛЖНО):** нормализация переносов строк в golden/serializer;
  использование `path`-абстракций (без хардкода `\`/`/`); отсутствие зависимостей от локали.
- **E2E-браузер:** запускается в Linux-контейнере CI и локально в WSL; на Windows — опционально.
- **Локальный DX:** melos-скрипты (`analyze`, `test`, `test:coverage`, `format`, `audit`,
  `serve`, `build:ssg`) работают одинаково в PowerShell (Windows) и bash (WSL).

### 11.5 Инструменты качества

- `dart analyze` со строгим `analysis_options` (lints), `dart format` в CI (проверка).
- Conventional Commits + melos `version`/`publish` (dry-run по умолчанию).
- Обязательные проверки границ зависимостей (core без Flutter, server без Flutter).

## 12. Дорожная карта фаз

> Детальная декомпозиция каждой фазы делается отдельно. Здесь — верхнеуровневый план и порядок.

- **Phase 0 — Foundation.** Monorepo (melos + pub workspaces), CI-матрица Windows/WSL,
  `analysis_options`, скелеты пакетов, соглашения (ADR, Conventional Commits), политика версий Flutter.
- **Phase 1 — Core модель и сериализатор (TDD).** `DocumentNode`, метаданные, HTML-сериализатор,
  escaping/санитайз, golden/property-тесты. **Веха: детерминированный безопасный HTML из модели.**
- **Phase 2 — SEO-артефакты.** JSON-LD, `sitemap.xml`, `robots.txt`, route/island-манифесты,
  валидаторы. **Веха: полный SEO-набор из ядра.**
- **Phase 3 — Widgets (двойная природа) + headless-извлечение.** `Seo.*`, `Island`, `HydraApp`,
  извлечение `DocumentNode` из Flutter-дерева. **Веха: HTML из реального Flutter-дерева в тестах.**
- **Phase 4 — SSG (`hydraline_build`).** Обход маршрутов, `dist/`, динамические сегменты,
  совместимость со статик-хостингами. **Веха: статический сайт с настоящим HTML + sitemap.**
- **Phase 5 — Server (`hydraline_server`).** shelf + Dart Frog адаптеры, per-request билдеры,
  статусы/редиректы/noindex, кэш, отдача sitemap/robots. **Веха: динамический SSR-маршрут.**
- **Phase 6 — Гидрация островов.** Island-загрузчик (JS-interop), multi-view монтирование,
  директивы (`load/idle/visible/media/manual`), anti-CLS, E2E в браузере. **Веха: рабочий hybrid-маршрут.**
- **Phase 7 — DevTools/Audit.** CLI-аудит «что видит краулер», SEO-валидации, dev-оверлей,
  проверка рассинхрона SSG↔hydrated. **Веха: CI-гейт по SEO/гидрации.**
- **Phase 8 — Полировка, доки, публикация.** Примеры (blog/docs/landing/product), гайды по
  хостингам, бенчмарки, публикация на pub.dev.
- **Phase 9 (МОЖЕТ) — Продвинутое.** Server islands (`server:defer`-аналог), стриминг,
  инкрементальная SSG, `hydraline_lint`.

**Порядок обязателен:** каждая фаза начинается только после выполнения Definition of Done
предыдущей (§13). Каркасы и код — по отдельным ТЗ на фазу.

## 13. Критерии приёмки (Definition of Done)

### 13.1 DoD уровня фичи/фазы (ДОЛЖНО)

1. Есть design-note/ADR (brainstorm) с зафиксированным решением.
2. Публичный API заморожен и задокументирован (dartdoc).
3. TDD-история соблюдена (тесты предшествуют коду; видно в истории).
4. Покрытие ≥ порога (§11.3), CI зелёный на Windows и WSL/Linux.

### 13.2 Приёмочные сценарии продукта (E2E, ДОЛЖНО)

- **A1.** Для `document`-маршрута `curl`/`view-source` возвращает валидный HTML с корректными
  title/meta/og/JSON-LD/заголовками/ссылками **без исполнения JS**.
- **A2.** Соц-превью (OG/Twitter) корректно строится симулятором бота (аудит) — теги в исходнике.
- **A3.** `hybrid`-маршрут: статический контент виден сразу; отмеченный остров гидрируется по
  своей директиве и становится интерактивным; CLS ≈ 0.
- **A4.** Существующее `app`-приложение продолжает работать **без изменений** после подключения
  hydraline (аддитивность).
- **A5.** `sitemap.xml` и `robots.txt` валидны и соответствуют route-манифесту.
- **A6.** SSR отдаёт корректные статусы/редиректы/`noindex`.
- **A7.** No-JS: страница остаётся осмысленной и навигируемой.
- **A8.** Один и тот же контент отдаётся боту и пользователю (нет cloaking).

## Приложение B. Глоссарий

- **App-shell** — минимальный HTML-каркас без контента, загружающий SPA.
- **CanvasKit / Skwasm** — WebGL/WASM-рендереры Flutter Web.
- **Cloaking** — отдача разного контента боту и пользователю; наказуемо поисковиками; **запрещено**.
- **CLS** — Cumulative Layout Shift (метрика сдвигов макета).
- **Hydration** — оживление pre-rendered HTML клиентским кодом (здесь — монтирование Flutter-острова).
- **Island** — изолированная интерактивная зона на статической странице.
- **SSG / SSR** — Static Site Generation (build-time) / Server-Side Rendering (request-time).
- **`DocumentNode`** — узел абстрактного дерева семантического документа (ядро hydraline).
- **`WebRouteRenderMode`** — режим маршрута: `app` / `document` / `hybrid`.
