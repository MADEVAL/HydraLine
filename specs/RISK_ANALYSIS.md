## R1. Регрессия multi-view sizing в Flutter 3.41.x (#185034)

### Конкретный сценарий

Владелец сайта обновляет Flutter с 3.38 до 3.41. `hybrid`-страница товара:
статический HTML-контент (описание, цена) отображается корректно, но остров
«калькулятор рассрочки» (Flutter) рендерится в canvas размером **не совпадающим**
с host-контейнером. Визуально: калькулятор либо обрезан, либо сжат в угол,
либо растянут за пределы контейнера. Интерактив (кнопки) работает, но hit-testing
смещён — пользователь нажимает не туда.

### Триггерные условия

- Flutter SDK ∈ [3.41.0, 3.41.x].
- `multiViewEnabled: true` + `app.addView({ hostElement })`.
- Несколько Flutter-view на странице (≥ 2 островов).
- Host-контейнеры имеют разные размеры (vw/vh, flex, проценты).
- **Не** переданы явные `viewConstraints` в `addView`.

### Статус фикса (проверено 2026-07-13)

- Issue #185034 — открыт, воспроизводится на stable и master.
- PR #185622 («Lock approach»), #185624 («State isolation»), #185698 («Minimum size»)
  от **mdebbar** — в процессе ревью, не вмёржены.
- Дополнительно: PR #184961 (holzgeist) — фикс рендера custom-element размера,
  связанный с canvas size > 16384px — **open**, не вмёржен.
- Корень: состояние гонки при инициализации/ресйнхронизации размеров нескольких
  view; последний добавленный view «перезаписывает» sizing-логику для всех.
- Дополнительный root cause (#175892): `OffscreenCanvas` имеет браузерный лимит
  размера (4096–16384px в зависимости от GPU/браузера). При `maxHeight: Infinity`
  canvas обрезается.

### Метрики порога

- Разница `canvas.width/height` vs `hostElement.getBoundingClientRect()` > 1px → FAIL.
- Количество задетектированных mismatches за прогон E2E > 0 → FAIL.

### Углублённый митигинг

1. **Матрица версий + CI-детектор (автоматический):**
   - CI содержит E2E-тест: создаётся страница с 3 островами разных размеров
     (фикс. px, vw, flex), делается скриншот каждого canvas и сравнение с host.
   - При обнаружении регрессии на новой версии Flutter — CI-блок (fail fast),
     автоматический issue с логом.
2. **Workaround первого выбора:** передача явных `viewConstraints` в `addView`:
   ```js
   app.addView({
     hostElement: el,
     viewConstraints: { maxWidth: el.clientWidth, maxHeight: el.clientHeight }
   });
   ```
   Документируется как recommended practice для всех версий ≥ 3.41.
3. **Резервный workaround (если constraints не помогают):** замена единого tall-view
   на несколько меньших view + IntersectionObserver (см. рекомендацию Flutter-team
   в #175892). Автоматизируется хелпером в `hydraline_widgets`.
4. **Пин версии:** в `pubspec.yaml` hydraline_widgets указать `flutter: ">=3.22.0 <3.41.0"`
   до полного исправления регрессии? **Нет** — это заблокирует пользователей на 3.44.
   Вместо этого: CI тестирует на 3.41 с ожидаемыми warning'ами; в документации —
   красный banner «Known issue with Flutter 3.41.x».
5. **План отступления:** если регрессия не исправлена к моменту широкого adoption,
   «островной» режим временно ограничивается одним островом на страницу
   (single-view fallback), что не подвержено багу.

### Верификация

- `hydraline_devtools` CLI: `hydraline audit --check-sizing` проверяет
  `canvas.getBoundingClientRect()` vs host для каждого острова.
- Golden-тесты эталонного рендера островов на поддерживаемых версиях Flutter.
- Ежемесячный ручной прогон на latest Flutter beta.

### Путь эскалации

Если workaround неэффективен и регрессия persists:
1. Открыть новый issue в flutter/flutter с минимальным reproduction на основе hydraline.
2. Временно снизить ценность multi-island до одного острова на страницу
   (ограничение документируется).
3. Активно участвовать в ревью PR'ов mdebbar'а.

## R2. Headless-извлечение DocumentNode окажется хрупким

### Конкретный сценарий

SSG запускается в CI. `flutter_tester` инициализирует биндинг, прогоняет
`pumpWidget()` с деревом, содержащим `Seo.text`, `Seo.image`, `Island`.
Но:
- **(a)** Какой-то `Seo.*`-виджет не попадает в обход `buildOwner` (например,
  лениво строится через `Builder`/`FutureBuilder`, который не резолвится в headless).
- **(b)** `pump()` не завершается из-за бесконечной анимации или незавершённого
  `Future` в дереве.
- **(c)** Platform channels (SystemChannels) кидают исключения, потому что
  headless-биндинг их не обрабатывает.

### Триггерные условия

- Виджет использует `MediaQuery`, `Navigator`, `Theme`, `Scaffold` — контекст,
  требующий ancestor-виджетов, которых нет в изолированном SSG-дереве.
- Виджет делает `HTTP-запрос` / `Future` в `initState`/`build`.
- Присутствуют платформенные плагины (камера, геолокация).
- Бесконечная анимация (`AnimatedBuilder`, `Ticker`) без `pump()`-контроля.

### Метрики порога

- SSG-прогон для эталонного проекта из ≥ 10 страниц проходит без exception за < 60 сек.
- Извлечённый `DocumentNode` содержит ≥ 90% ожидаемых узлов (сравнение с эталоном).

### Углублённый митигинг

1. **Spike Phase 3 (первый шаг, до архитектурного решения):**
   - Создать тестовый проект с полным спектром `Seo.*`-виджетов, разными
     глубинами вложенности, `InheritedWidget`-зависимостями.
   - Прогнать `flutter test` с извлечением. Задокументировать, **какие именно**
     паттерны не работают.
2. **Sandbox-виджет для SSG:**
   - `HydraPage` / `SsgPage` — обёртка, которая в headless-режиме подменяет
     проблемные зависимости на заглушки (напр. `MediaQuery` с фикс. размером,
     `Navigator` с mock-состоянием).
   - Это **не** фреймворк, а утилита для build-time контекста.
3. **Поверхность (B) как независимый fallback:**
   - Если поверхность (A, Flutter-виджеты) буксует для сложных страниц,
     разработчик всегда может перейти на pure-Dart `DocumentNode` builder.
   - Обе поверхности дают идентичный результат — миграция не ломает вывод.
4. **Градуальная стратегия:**
   - Phase 3: простые страницы (текст, изображения, ссылки) — 100% извлечение.
   - Phase 4-6: сложные страницы (вложенные острова, анимации) — gradual.
   - Документировать «SSG-compatible widget patterns».
5. **Отладка:**
   - `hydraline_devtools` показывает diff: «какие виджеты не попали в DocumentNode».
   - Режим `--verbose` логирует каждый посещённый/пропущенный виджет.

### Верификация

- Golden-тесты `DocumentNode` из Flutter-дерева vs pure-Dart билдера (должны совпадать).
- SSG-прогон на репрезентативном наборе страниц (блог, лендинг, товар, документация).
- Тест на устойчивость: 100 последовательных SSG-прогонов без деградации результата.

### Путь эскалации

Если извлечение нестабильно для широкого класса страниц:
1. Поверхность (A) маркируется как «beta, требует соблюдения SSG-паттернов».
2. Поверхность (B, pure-Dart) становится рекомендуемой для production SSG.
3. Документируется как явное ограничение, а не как баг.

## R3. Стоимость старта движка бьёт по UX

### Конкретный сценарий

Пользователь (человек, не бот) заходит на `hybrid`-страницу. HTML (текст, meta)
отображается мгновенно. Но Flutter-остров (калькулятор, форма) остаётся серым
скелетоном **3–8 секунд** — пока грузится `main.dart.js` (1.5–2.5 MB), затем
`canvaskit.wasm` (≈1.5 MB brotli), затем `flutter_bootstrap.js`, затем
инициализируется движок, затем загружается код острова, затем `addView`
монтирует Flutter-view в DOM.

На 3G: **8–15 секунд** до интерактивности. Пользователь уходит (bounce rate +32%
при задержке 1→3 сек, данные Google).

### Триггерные условия

- Холодный кэш браузера (первый визит).
- Медленное соединение (3G/4G с плохим покрытием).
- Несколько островов на странице → дополнительная задержка per-view инициализации.
- Тяжёлый `main.dart.js` (много бизнес-логики в основном бандле).
- Остров использует директиву `hydrateOnLoad` (eager) и находится «выше сгиба».

### Реальные цифры (проверено)

| Ресурс / сценарий | Холодный кэш (3G) | Тёплый кэш | Холодный кэш (4G) |
|---|---|---|---|
| `flutter_bootstrap.js` + `flutter.js` | ~18 KB (inline) | 0 | 0 |
| `main.dart.js` (minimal app, brotli) | ~600 KB / ~3 сек | 0 | ~1 сек |
| `main.dart.js` (real app, brotli) | ~1.5–2.5 MB / ~8–12 сек | 0 | ~3–5 сек |
| `canvaskit.wasm` (chromium, brotli) | ~1.1 MB / ~5 сек | 0 (CDN) | ~1.5 сек |
| Весь Flutter engine init | ~1–2 сек | ~0.5 сек | ~1 сек |
| **Итого: первый байт → интерактивный остров** | **10–19 сек** | **~1 сек** | **4–7 сек** |

### Метрики порога

- TTI (Time To Interactive) острова при холодном кэше 4G: < 5 сек (target).
- LCP (Largest Contentful Paint): HTML-контент < 2.5 сек.
- Lighthouse Performance score: ≥ 70 для `hybrid`-страницы.

### Углублённый митигинг

1. **Директивы — первый рубеж:**
   - **По умолчанию остров = `hydrateOnIdle`**, а не `hydrateOnLoad`.
     Исключение — только явный выбор разработчика.
   - `hydrateOnVisible` для контента «ниже сгиба»: движок не загружается,
     пока пользователь не доскроллил.
   - Это радикально снижает TTI для критического контента (HTML уже виден).
2. **Скелетоны и anti-CLS:**
   - В HTML встраивается скелетон острова с зарезервированными размерами
     (полученными из SSR/SSG-прохода).
   - Скелетон визуально «похож» на будущий Flutter-контент (серые блоки),
     чтобы пользователь не видел «прыжков».
3. **Prefetch движка (опционально):**
   - `<link rel="preload" href="main.dart.js" as="script">` + аналогично для wasm.
   - `<link rel="modulepreload">` для ES-модулей.
   - Генерируется build-инструментом (`hydraline_build`).
4. **Прогрессивная загрузка движка:**
   - Сначала HTML (TTFB < 100ms, FCP < 1 сек).
   - Затем `flutter_bootstrap.js` (inline) — мгновенно.
   - Затем `main.dart.js` — в фоне, без блокировки рендера.
   - Затем `canvaskit.wasm` — параллельно с `main.dart.js` (preload).
   - Затем инициализация движка — только когда JS исполнен.
   - Затем острова — по директивам.
5. **Разделение кода (code splitting):**
   - Deferred imports (`deferred as`) для тяжёлых частей приложения.
   - Островной код выделяется в отдельный `part.js`, загружаемый только при
     триггере острова.
6. **Бюджет размера (документированный):**
   - Островной bundle (сам остров + его уникальные зависимости): < 100 KB gzip.
   - Если остров тяжелее — предупреждение devtools + рекомендация code-split.
   - Скелетонный HTML-контент страницы: < 50 KB gzip.

### Верификация

- E2E-тест: замер TTI острова в headless Chrome (Lighthouse CI в CI-пайплайне).
- Бенчмарк: холодный кэш 4G (Lighthouse throttling) для эталонной `hybrid`-страницы.
- Регрессионный тест: размер `main.dart.js` не растёт > 5% между версиями hydraline.

### Путь эскалации

Если TTI > порога после всех оптимизаций:
1. Документировать: «Flutter Web не предназначен для мгновенной интерактивности
   на холодном кэше; для instant-interactive используйте нативный JS».
2. Рекомендовать progressive enhancement: некритичные острова — `hydrateOnVisible`;
   критичные — чистый JS, не Flutter.
3. В долгосрочной перспективе — инвестировать в снижение стартового бандла
   (ждём улучшений от Flutter-команды: wasm tree-shaking, engine splitting).

## R4. Нативная поддержка #187663 «догонит»

### Конкретный сценарий

Flutter-команда реализует `WebRouteRenderMode` и DocumentNode в ядре (2027+).
Hydraline становится дублирующим функционалом. Пользователи спрашивают:
«Зачем hydraline, если Flutter теперь умеет сам?»

### Оценка вероятности и сроков

- #187663 — open, с дизайн-доком, назначен на web-команду. Но:
- #46789 (610 👍) открыт с 2019 (6 лет) — P2, не приоритет.
- #139292 (DOM renderer) — accepted 2024, низкий приоритет.
- Flutter-команда явно говорит: «делайте как community package» (#183992).
- Фокус Flutter Web сейчас — Skwasm/CanvasKit унификация (#175624) и
  multi-window для desktop, а не SEO/DOM.
- **Оценка:** нативная поддержка — 2028+ при самых оптимистичных сроках.
  Hydraline имеет окно ≥ 2 лет как единственное решение.

### Углублённый митигинг

1. **Совместимость модели по контракту:**
   - Термины и API hydraline (`WebRouteRenderMode`, `DocumentNode`, island manifest)
     осознанно совпадают с дизайн-доком #187663.
   - При появлении нативной поддержки миграция — замена импорта `hydraline` →
     `flutter_web` с адаптацией, а не переписывание контента.
2. **Стратегия «надстройка, не замена»:**
   - Hydraline позиционируется как **реализация дизайн-дока #187663 для текущих
     версий Flutter**, а не как конкурент ядру.
   - Когда нативная поддержка появится, hydraline становится тонким адаптером
     + инструментарием поверх неё.
3. **Дифференциация, которую ядро не даст:**
   - DevTools-инспектор «что видит краулер» — маловероятно в ядре.
   - CLI-аудиты для CI — маловероятно в ядре.
   - Специфические хостинг-адаптеры (Firebase, Netlify, Cloudflare Pages).
   - Глубокая интеграция с pure-Dart server-билдерами.

### Верификация

- Регулярный обзор изменений в #187663 и Flutter web roadmap (ежеквартально).
- Тест совместимости: эталонный `DocumentNode` hydraline должен быть
  конвертируем в гипотетический нативный DocumentNode за O(N).

## R5. Ошибки экранирования → XSS

### Конкретный сценарий

Пользовательский контент (имя, комментарий, описание товара) содержит
`<script>alert('xss')</script>`. HTML-сериализатор вставляет его без
экранирования → JavaScript исполняется в браузере жертвы.

Или: URL картинки содержит `" onerror="alert(1)` при вставке в атрибут
без escaping → attribute-based XSS.

Или: `javascript:alert(1)` в `href` ссылки, построенной из пользовательского
ввода → скрипт исполняется при клике.

### Триггерные условия

- Пользовательский ввод попадает в `DocumentNode.text()` без escaping.
- Пользовательский ввод попадает в `href`/`src` атрибуты без URL-санитайза.
- `UnsafeHtmlNode` используется без санитайзера.
- Сериализатор не экранирует спецсимволы (`<`, `>`, `&`, `"`, `'`) в контексте.

### Метрики порога

- **Property-тесты:** 0 XSS за 10^6 случайных входов.
- **Fuzz-тесты:** 0 падений/ошибок за 60 сек фаззинга.
- Статические анализаторы (dart analyze + собственные линты) — 0 warning'ов
  на тестовом наборе.

### Углублённый митигинг

1. **Контекстное экранирование (context-aware):**
   - `escapeHtmlText(str)` — для текстового контента: `<` → `&lt;`, `>` → `&gt;`,
     `&` → `&amp;`.
   - `escapeHtmlAttribute(str)` — для значений атрибутов: `"` → `&quot;`, `'` → `&#x27;`.
   - `sanitizeUrl(str)` — allowlist схем: `http`, `https`, `mailto`, `tel`. Блок:
     `javascript:`, `data:`, `vbscript:`.
   - Отдельные функции → нельзя случайно применить не ту.
2. **Безопасный API по умолчанию:**
   - `DocumentNode.text(content)` **всегда** экранирует контент.
   - `DocumentNode.link(href: url)` — требует `SafeUrl` (результата `sanitizeUrl`).
   - `UnsafeHtmlNode` — единственный способ вставить сырой HTML. Имя «Unsafe»
     как постоянное напоминание.
3. **Property-based тестирование (TDD):**
   - Генерация случайных строк (включая юникод, escape-последовательности,
     известные XSS-векторы).
   - Проверка: после `serialize → parse browser DOM → извлечь текст` =
     исходный текст, без исполнения скриптов.
   - Используется библиотека `glados` или `rapid` для property-тестов.
4. **Fuzz-тестирование:**
   - Вход: случайные байтовые последовательности.
   - Проверка: сериализатор не падает, не производит невалидный HTML,
     не содержит `<script>` без экранирования.
5. **`hydraline_lint` (поздние фазы):**
   - Правило: «`UnsafeHtmlNode` без санитайзера» → warning.
   - Правило: «пользовательский ввод в `href` без `sanitizeUrl`» → warning.
6. **CSP (Content Security Policy) в документации:**
   - Рекомендуемый CSP для продакшна: `script-src 'self' 'wasm-unsafe-eval';`
   - Без `'unsafe-inline'` — блокирует инлайн-скрипты, даже если XSS прошёл.

### Верификация

- Пайплайн CI: `dart test --tags security` с property-тестами и фаззингом.
- Ручной пентест перед каждым мажорным релизом с использованием OWASP-векторов.
- Статический анализ кодовой базы на наличие `UnsafeHtmlNode` без санитайзера.

### Путь эскалации

При обнаружении XSS-уязвимости:
1. Security-фикс выпускается в течение 48 часов (patch-версия).
2. Публикуется security advisory (GitHub → Security → Report).
3. Добавляется регрессионный тест на конкретный вектор.

## R6. Рассинхрон SSG-HTML и гидрированного DOM

### Конкретный сценарий

SSG генерирует `<h2>Старая цена: 1000₽</h2>`. Остров «калькулятор цены»
гидрируется и перерендеривает ту же семантическую зону, но с новыми данными:
`<h2>Цена: 950₽ (скидка 5%)</h2>`. Пользователь видит двойной контент,
«прыжок» при гидрации, или рассинхрон между статическим HTML и Flutter-рендером.

Хуже: статический HTML — ссылка `<a href="/old">`, а гидрированный Flutter —
`<a href="/new">`. Краулер проиндексировал `/old`, а пользователи кликают `/new`.
Рассинхрон URL-пространства.

### Триггерные условия

- SSG-прогон сделан на устаревших данных (товар изменился, страница пересобрана).
- Контент генерируется из разных источников для SSG и для острова
  (SSG: статичный JSON; Flutter-остров: fetch с API).
- Изменение шаблона/структуры страницы между SSG-прогоном и деплоем.
- Остров рендерит семантический контент, а не только интерактив.
- Props острова в SSG-манифесте ≠ props при гидрации.

### Метрики порога

- DevTools-аудит: сравнение текстового содержимого SSG-HTML и post-hydration DOM.
  Расхождение > 5% текстовых узлов → warning.
- E2E-тест: визуальная регрессия «до гидрации» vs «после» (скриншот) — разница
  только в интерактивных зонах.

### Углублённый митигинг

1. **Единый источник контента (Single Source of Truth):**
   - И SSG, и остров получают данные из **одного источника** на момент сборки.
   - Для SSG: данные замораживаются в момент `flutter build`.
   - Для острова: те же данные передаются как `initialData` через манифест.
   - Документируется: не делайте fetch в `initState` острова, если контент
     уже есть в статическом HTML.
2. **DevTools-проверка «SSG vs Hydrated»:**
   - После гидрации сравнивает DOM-дерево: текстовые узлы, атрибуты ссылок,
     alt-тексты изображений.
   - Подсвечивает расхождения в оверлее (зелёный = совпадает, красный = рассинхрон).
3. **Golden-тесты на оба состояния:**
   - Golden №1: SSG-HTML (до гидрации) — как эталон.
   - Golden №2: post-hydration DOM (после `pump()`) — как эталон.
   - Тест: семантически значимые части (заголовки, ссылки, текст) должны совпадать.
4. **Принцип прогрессивного улучшения (строгий):**
   - Остров **дополняет** статический контент интерактивом, но **не заменяет** его.
   - Если острову нужно показать динамику (живую цену) — он показывает её
     **добавочно** к статической, а не вместо.
   - Статический контент остаётся нетронутым. Это исходная идея islands.

### Верификация

- CI-тест: аудит на рассинхрон для всех example-страниц.
- Ручная проверка devtools-оверлея на каждом PR в widgets/server/build.

### Путь эскалации

При обнаружении систематического рассинхрона:
1. Временно требовать явного указания `replace: true` в `Island` для зон,
   где гидрированный контент заменяет статический.
2. Добавить в SSR-ответ предупреждающий комментарий `<!-- hydraline: replaced-by-island -->`.

## R7. Cloaking-соблазн

### Конкретный сценарий

Разработчик сталкивается с тем, что SSR/SSG для динамических маршрутов сложен
(pure-Dart билдер, интеграция с БД). Смотрит на статью krootl.com («Firebase
Function отдаёт HTML только ботам») и думает: «Сделаю так же — проще».
Результат:
- Бот (Googlebot/Facebook) получает полный HTML с тегами.
- Пользователь получает app-shell.

Это **cloaking** — нарушение Google Webmaster Guidelines. Санкция: исключение
из индекса (de-indexing) или ручные меры (manual action).

### Триггерные условия

- UA-сниффинг в серверном коде.
- Разный HTML для ботов и пользователей.
- Отсутствие понимания, почему cloaking опасен.

### Метрики порога

- Архитектурный запрет: **не существует** метода в API hydraline, который
  принимает `userAgent` и меняет вывод в зависимости от него.
- Тест: `DocumentNode → сериализовать → DocumentNode → сериализовать` —
  детерминизм (одинаковый результат).

### Углублённый митигинг

1. **Архитектурный барьер (дизайн API):**
   - `hydraline_server` middleware не имеет доступа к `User-Agent`.
   - Route handler получает `(request, data)`, но `DocumentNode` builder
     не знает, кто запрашивает. Сигнатура: `DocumentNode Function(Request, T data)`.
2. **Образовательный барьер:**
   - Первая страница документации: «Hydraline **не** делает cloaking. Вот почему
     это опасно: (ссылки на Google guidelines, issue #46789 с предупреждением)».
   - Примеры: показывать, как делать динамический SSR **правильно** (pure-Dart
     билдер + единый код для всех).
3. **DevTools-аудит:**
   - Проверка: `curl <url>` и `curl -H "User-Agent: Googlebot" <url>` → HTML
     должен быть идентичен (кроме динамических meta, которые законны для всех).
   - CI-гейт: если аудит находит разный HTML — fail.

### Верификация

- CI-тест: для каждого example-маршрута сравниваются ответы с разными UA —
  должны быть идентичны.
- Приёмочный сценарий A8 (ТЗ §13.2).

### Путь эскалации

Если cloaking обнаружен в production-использовании:
1. Это не баг hydraline, а misuse. Документируется.
2. Добавляется детектор cloaking в devtools с громким предупреждением.

## R8. Разные ОС (CRLF/пути) ломают golden-тесты

### Конкретный сценарий

Golden-тест HTML-сериализатора проходит на WSL (Linux, `\n`), но падает на
Windows (PowerShell, `\r\n` в git checkout). Или наоборот: разработчик на
Windows создаёт golden с `\r\n`, CI на Linux падает.

Или: пути в golden-данных `/home/runner/...` vs `C:\Users\...` — тесты
непереносимы.

### Триггерные условия

- Windows-раннер CI использует `git config core.autocrlf true`.
- Golden-файлы закоммичены с CRLF.
- В выводе сериализатора используются платформенные пути.
- Разные кодировки по умолчанию (UTF-8 vs UTF-8 with BOM).

### Метрики порога

- Все golden-тесты проходят на Windows **и** Linux/WSL CI.
- Ноль флаки-тестов из-за CRLF/путей за последние 30 дней.

### Углублённый митигинг

1. **Нормализация переносов строк в рантайме:**
   - HTML-сериализатор всегда выводит `\n` (Unix), независимо от ОС.
   - При сравнении golden: ожидаемое значение нормализуется (`\r\n` → `\n`).
2. **`.gitattributes` в корне репозитория:**
   ```gitattributes
   # Все текстовые файлы — LF в репозитории, независимо от ОС
   * text=auto eol=lf

   # Golden-файлы — бинарные (без конвертации)
   **/test/**/goldens/** binary
   ```
3. **Абстракция путей:**
   - Все пути в тестах используют `package:path` и `path.posix` для
     сравнения (форвард-слеши).
   - Отсутствие хардкода `\` или `/`.
4. **CI-матрица:**
   - Golden-тесты запускаются на Windows-runner и Ubuntu-runner.
   - При падении на одной ОС — блокирующая ошибка.
5. **Локальное удобство для Windows-разработчиков:**
   - `melos run test:golden` нормализует переносы строк и логирует
     предупреждение, если git config настроен некорректно.

### Верификация

- Еженедельный Scheduled CI-прогон golden'ов на обеих ОС.

## R9. Ограничение «сервер без Flutter» неверно понято

### Конкретный сценарий

Разработчик читает «hydraline_server рендерит HTML на сервере» и ожидает, что
можно написать Flutter-виджеты (с `Seo.text`, `Island` и Material-кнопками),
задеплоить на Dart Frog-сервер, и они будут работать как SSR. Это невозможно:
Dart Frog не имеет `dart:ui`, `CanvasKit`, rendering pipeline.

Разработчик разочарован, оставляет негативный отзыв: «hydraline не умеет SSR,
только SSG».

### Триггерные условия

- Поверхностное чтение документации.
- Ожидание, что «SSR» в названии = серверный рендер виджетов.
- Отсутствие понимания архитектуры Flutter (dart:ui, движок, CanvasKit).

### Метрики порога

- Количество issues/вопросов «почему SSR не работает с Flutter-виджетами» → 0
  после улучшения документации.
- Документация явно объясняет различие до первого примера кода.

### Углублённый митигинг

1. **Первая страница README (жирным):**
   > ## Hydraline SSR рендерит HTML, но **не исполняет Flutter-виджеты на сервере**.
   > Для динамического SSR используйте pure-Dart `DocumentNode` builder.
   > Flutter-виджеты работают в SSG (build-time) и в браузере (runtime islands).
2. **Диаграмма в документации:**
   - Визуальное разделение: «SSG (build-time, есть Flutter)» vs «SSR (request-time, pure Dart)».
   - §5.3 ТЗ — must-read перед использованием.
3. **API-дизайн как подсказка:**
   - Импорт `package:hydraline_widgets` не компилируется в pure-Dart проекте
     (зависимость от Flutter) — ошибка компиляции сразу подсказывает неверное использование.
4. **Примеры:**
   - SSR-пример: Dart Frog route + pure-Dart `DocumentNode` builder.
   - SSG-пример: `flutter build` + `hydraline_build`.
   - Hybrid-пример: SSG HTML + Flutter-остров в браузере.

### Верификация

- Обратная связь от ранних пользователей.
- Анализ типов issues на GitHub (категоризация).

### Путь эскалации

При сохранении путаницы:
1. Переименовать «SSR» в «Dynamic HTML Rendering (pure Dart)».
2. Сделать видеотуториал «Hydraline SSR vs SSG: когда что использовать».
3. Добавить `analysis_options` правило для Dart Frog-проектов: запрет импорта
   `package:flutter` (предлагается сообществу Dart Frog).
