## Q1. Headless-биндинг для извлечения DocumentNode из Flutter-дерева

### Контекст

Для поверхности авторинга (А) нужно извлечь `DocumentNode` из Flutter-виджетов
(`Seo.text`, `Island` и т.д.) в среде, где `dart:ui` доступен, но GPU **нет**
(CI, build-сервер). Варианты:

- **(V1)** `flutter_tester` — то, на чём работает `flutter test`.
- **(V2)** Кастомный `WidgetsFlutterBinding` поверх Platform Dispatcher.
- **(V3)** Отдельный Dart-процесс, запускающий Flutter engine в headless-режиме.

### Анализ

**V1: `flutter_tester` / `AutomatedTestWidgetsFlutterBinding`**
- Это **существующий, стабильный** биндинг, на который полагается весь `flutter test`.
- Использует `FakeAsync`-зону (детерминированное время), `buildOwner`, `pipelineOwner`
  и полный rendering pipeline без GPU.
- Вызывается через `testWidgets()` или напрямую `AutomatedTestWidgetsFlutterBinding.ensureInitialized()`.
- Позволяет `pumpWidget()`, `pump()`, обход `buildOwner`/`Element`-дерева.
- Минус: завязан на `package:flutter_test`, который тянет много зависимостей.
- Не поддерживается в web-среде (но для SSG web и не нужен).

**V2: Кастомный биндинг**
- Технически возможно: создать наследник `WidgetsFlutterBinding` + `TestPlatformDispatcher`.
- Требует тонкой настройки: регистрация каналов (SystemChannels, Lifecycle, Platform),
  обеспечение `SchedulerBinding`, `RendererBinding`, `SemanticsBinding`.
- `flutter_tester` уже делает это — V2 был бы переизобретением.

**V3: Отдельный процесс**
- Запуск `dart` с `dart:ui` (требует Flutter engine, а не просто Dart VM) — по сути
  равен запуску `flutter test` в подпроцессе.
- Оверхед IPC/сериализации.

### Решение

**Выбран V1 (`flutter_tester` через `AutomatedTestWidgetsFlutterBinding`).**

Конкретно:
1. `hydraline_build` использует `flutter test` как среду исполнения (не делает
   IPC в отдельный процесс, а **исполняется внутри** `flutter test`).
2. Инструмент SSG запускается как **тест-скрипт**: `dart run hydraline_build:generate`
   внутри проекта, который через `flutter test --tags ssg` запускает headless-извлечение.
3. Альтернативно: `hydraline_build` — это исполняемый файл, скомпилированный **под Flutter**
   (не pure Dart), инициализирующий биндинг вручную `AutomatedTestWidgetsFlutterBinding.ensureInitialized()`.

Обоснование:
- `flutter_tester` — стабильный, документированный, поддерживаемый Flutter SDK.
- Нет нужды поддерживать собственный биндинг и чинить его при каждом обновлении Flutter.
- Известные ограничения (нет платформенных плагинов, web-специфики) **не важны** для SSG
  (нам не нужны камеры/геолокация/CanvasKit в headless).
- Сообщество уже использует `flutter test` для генерации golden-файлов и скриншотов —
  извлечение `DocumentNode` той же природы.
- Spike в Phase 3 проверит, что обход `buildOwner` / `Element`-дерева даёт полную
  информацию о `Seo.*`-виджетах.

### Альтернатива на будущее

Если `flutter_tester` окажется слишком тяжёлым для CI (холодный старт), можно выполнить
извлечение **один раз** и закэшировать `DocumentNode` как артефакт сборки
(промежуточный `.hydraline.json`). Тогда повторные сборки без изменений Flutter-дерева
обходится без `flutter_tester`.

## Q2. Формат route-манифеста

### Контекст

Route-манифест описывает: путь → режим (`app|document|hybrid`), метаданные, статус,
источник контента, canonical. Варианты формата:

- **(V1)** Декларативный `.yaml` / `.json` файл рядом с `pubspec.yaml`.
- **(V2)** Код-конфиг (Dart-функция/класс в `lib/routes.dart`).
- **(V3)** Аннотации на маршрутах / виджетах (как `@RoutePage()` в `auto_route`).
- **(V4)** Автогенерация из роутера (извлечение `GoRoute`-дерева).

### Анализ

**V1: Статический YAML/JSON**
- Плюс: максимально прост, не требует компиляции, читается сервером без Flutter.
- Плюс: легко редактируется, diff-friendly, валидируется схемой.
- Минус: рассогласование с роутером (два источника правды).

**V2: Dart-код**
- Плюс: типизация, рефакторинг, автодополнение IDE.
- Плюс: можно встроить в `GoRouter`-конфиг как extension.
- Минус: требует Dart VM / компиляции для чтения сервером.

**V3: Аннотации**
- Плюс: «source of truth» рядом с кодом.
- Минус: code generation (build_runner), усложнение, не подходит для сервера.

**V4: Автогенерация из роутера**
- Плюс: zero-config для пользователя.
- Минус: из `GoRoute`/`auto_route` дерева можно извлечь только путь, но не метаданные,
  режим, источник контента. Нужны дополнительные аннотации → возврат к V3.

### Решение

**Гибрид V1+V2: декларативный файл (`hydraline.routes.yaml`) как primary source; Dart-код (builder) как secondary, генерирующий тот же файл.**

Конкретно:
1. Канонический формат: **YAML** (`hydraline.routes.yaml` в корне проекта).
2. Схема:
   ```yaml
   routes:
     - path: /
       mode: document
       metadata: { title: Home, description: ... }
       content_source: widget   # или dart_builder:HomeBuilder.new
     - path: /blog/:slug
       mode: document
       metadata: { title: Blog }
       content_source: dart_builder:BlogPostBuilder.new
       dynamic_segments: { slug: [post-1, post-2] }  # только для SSG
     - path: /app/dashboard
       mode: app
   ```
3. **Dart builder API** для программного создания того же YAML (типизированно):
   ```dart
   final manifest = HydraRoutes()
     ..document('/',
         meta: Meta(title: 'Home'),
         content: WidgetContent(HomePage.new))
     ..document('/blog/:slug',
         meta: Meta(title: 'Blog'),
         content: DartBuilderContent(blogPostBuilder),
         dynamicSegments: {'slug': slugs});
   // → сериализовать в YAML
   ```
4. Инструменты читают **YAML**. Разработчик **может** генерировать его из Dart или
   писать вручную. Два пути не конфликтуют.
5. **Не** делаем аннотации/автогенерацию из роутера в MVP (фаза огр. Phase 3+).

Обоснование:
- YAML читается сервером (pure Dart, без компиляции Flutter).
- YAML читается SSG-инструментом в headless-биндинге.
- Dart builder — опциональное удобство, не обязательство.
- Разделение роутера и SEO-манифеста — осознанное: роутер занимается навигацией,
  манифест — тем, «что отдать краулеру». Это разные задачи.
- Избегаем зависимости от конкретного роутера (Q3).

## Q3. Границы поддержки роутеров

### Контекст

В ТЗ W-4: «совместимость с популярными роутерами через адаптеры; не навязывать
конкретный роутер». Надо определить: кто first-class, кто best-effort.

### Данные (2026)

| Роутер | Статус | Поддержка | Популярность |
|---|---|---|---|
| `go_router` | Flutter team, feature-complete | активная, багфиксы | #1 (~105K загрузок/день) |
| `auto_route` | Community (Milad Akarie) | поддерживается, v11.1.0 | #2 (~10K/день) |
| `beamer` | Community (slovnicki) | v1.7.0, последний push окт 2024, **низкая активность** | #3 (~2K/день) |
| Raw Navigator 2.0 | Flutter SDK | всегда | n/a |

### Анализ

- **`go_router`** — официальный, самый распространённый. Предоставляет
  `RouteConfiguration.debugKnownRoutes()` и доступ к дереву маршрутов — это всё,
  что нужно для связки с route-манифестом.
- **`auto_route`** — мощный, но генерирует код. Имеет `AutoRouteInformationProvider`,
  дерево маршрутов доступно через конфиг.
- **`beamer`** — практически заморожен (последний push 2024, 125 открытых issues,
  Dart 3 совместимость «средняя»).
- **Navigator 2.0** — всегда доступен как fallback. Адаптер для него покрывает всё.
- Hydraline **не зависит** от роутера архитектурно (route-манифест — внешний YAML).
  Единственная точка касания роутера — момент, когда **build-time SSG обходит
  маршруты** для извлечения `DocumentNode`.

### Решение

**First-class: `go_router` + «адаптерный интерфейс» для всего остального.**

Конкретно:
1. **First-class адаптер** для `go_router` (Phase 3): helper для связки `GoRoute`-дерева
   с route-манифестом (сверка, предупреждения о расхождении). Бонус: `go_router` имеет
   `debugKnownRoutes()` для отладки.
2. **Абстрактный интерфейс** `RouteAdapter` с методами:
   - `List<RouteInfo> get routes` — все маршруты
   - `Future<void> navigateToForExtraction(RouteInfo route)` — для headless-обхода
3. **Встроенные реализации:**
   - `GoRouterAdapter` (first-class, в `hydraline_widgets`)
   - `Navigator2Adapter` (fallback, для сырого Navigator 2.0)
4. **`auto_route`** — best-effort через `RouteAdapter`-интерфейс. Hydraline **не**
   зависит от `auto_route` в своём pubspec. Сообщество может написать адаптер.
5. **`beamer`** — не поддерживаем (пакет практически мёртв). Пользователи beamer
   мигрируют на общий `Navigator2Adapter` или переходят на go_router.

Обоснование:
- `go_router` доминирует с огромным отрывом (>10× downloads над вторым местом).
- Абстрактный `RouteAdapter` даёт сообществу возможность поддерживать любой роутер
  без изменений в hydraline.
- `beamer` не оправдывает инвестиций (заморожен, малая аудитория).
- `auto_route` получает поддержку через интерфейс, не добавляя зависимость в hydraline.

## Q4. Богатство DocumentNode (таблицы, встраиваемый HTML)

### Контекст

C-1 определяет минимум узлов. Вопрос: насколько расширять в сторону сложных
структур (таблицы, вложенный HTML) в MVP.

### Анализ

**Минималистичный подход (только базовые блочные/инлайн-элементы):**
- h1–h6, p, ul/ol/li, a, img, section/article/nav/header/footer, blockquote, code/pre
- Достаточно для блогов, лендингов, документации.
- Безопасен по умолчанию.

**Средний подход (базовый + таблицы):**
- + table/thead/tbody/tr/th/td, caption, colgroup (ограниченно)
- Покрывает карточки товаров (таблицы спецификаций), сравнения, данные.
- Требует осторожного экранирования (ячейки не должны содержать произвольный HTML).

**Полный подход (raw-HTML-фрагменты, video, audio, iframe):**
- Открывает XSS и проблемы безопасности.
- Делает `DocumentNode` сложным и раздутым.
- Противоречит цели «developer-controlled semantics».

### Решение

**MVP (Phase 1): базовый набор узлов + таблицы (безопасный поднабор). Raw-HTML — opt-in.**

Конкретно:
1. **Phase 1 (MVP, модель `DocumentNode`):**
   - Обязательные: document, head/meta, title, h1–h6, p, text/span, a, img, ul/ol/li,
     section/article/nav/header/footer/main, blockquote, code/pre, time.
   - **Таблицы** (MVP): table, thead, tbody, tr, th, td. Только текст в ячейках
     (не inline-элементы, кроме a/img). Без colspan/rowspan в MVP.
   - Island-placeholder.
   - Примечание: весь набор узлов входит в Phase 1 (Core), т.к. это часть модели
     `DocumentNode` и сериализатора. SEO-артефакты (sitemap/robots/JSON-LD) — Phase 2.
2. **Phase 8+:**
   - Расширенные таблицы (colspan/rowspan, caption, colgroup).
   - figure/figcaption.
   - details/summary (no-JS-фоллбэк для аккордеонов).
3. **Raw-HTML:**
   - Только через `UnsafeHtmlNode(text, sanitizer?)`. По умолчанию санитайзер **не**
     встроен — разработчик передаёт свой или библиотечный.
   - Явное предупреждение в dartdoc, анализатор `hydraline_lint` предупреждает об
     использовании `UnsafeHtmlNode` без санитайзера.
4. **Не делаем никогда:**
   - iframe, script, video, audio (пусть разработчик использует `UnsafeHtmlNode`
     со своим санитайзером).
   - CSS/стили в `DocumentNode` (это семантическая модель, не презентационная).

Обоснование:
- 80% SEO-контента покрывается h1–h6 + p + a + img + списки.
- Таблицы — следующий по частоте запрос (сравнения, спецификации, цены).
- Безопасность важнее мощности; raw-HTML — мощный, но опасный инструмент, должен
  быть осознанным выбором (opt-in + санитайзер).
- Принцип P3: «безопасность по умолчанию» — дерево без raw-узлов гарантированно
  безопасно.

## Q5. Server-islands (`server:defer`-аналог) в MVP или позже

### Контекст

Astro вводит понятие **server island** — компонент, который рендерится сервером
отдельно от основной страницы (независимый запрос, параллельная загрузка, не
блокирует TTFB). Аналог в hydraline: остров, чей HTML генерируется не в основном
SSG/SSR-проходе, а подгружается асинхронно.

### Анализ

**Зачем нужны server-islands:**
- Персонализированный контент (профиль пользователя, корзина).
- Медленные источники данных, которые не должны блокировать FCP.
- Разное TTL кэширования для статики и динамики на одной странице.

**Почему не в MVP:**
- Усложняет архитектуру: нужен endpoint для отдачи HTML-фрагмента, протокол
  подгрузки/вставки, кэширование фрагментов отдельно от страницы.
- Для MVP достаточно двух путей доставки: чистый SSG (статический HTML) и чистый SSR
  (полный HTML на запрос). Их композиция — продвинутый кейс.
- Hydraline — не фреймворк; серверные острова требуют более глубокой интеграции
  с сервером приложения.
- Astro потратил годы на зрелость базовых островов перед server islands.

### Решение

**Отложить в Phase 9. MVP не включает server-islands.**

Если разработчику нужен динамический контент на статической странице в MVP:
- Вариант 1: Использовать `hybrid`-режим + SSR для страницы целиком (не дешёво, но работает).
- Вариант 2: Использовать `app`-остров, который сам фетчит данные в рантайме
  (не SEO для динамики, но SEO для статической оболочки сохранён).
- Вариант 3: Для Firebase/Netlify — настроить rewrite на Cloud Function для конкретного
  пути (как в статье krootl.com), но это cloaking-грань (осторожно).

Phase 9 вернётся к этому после стабилизации базовой архитектуры.

## Q6. Стратегия передачи props в остров

### Контекст

Island-плейсхолдер в HTML содержит `data-props` с сериализованными пропами.
Эти пропы пересекают сервер-клиентскую границу. Нужна стратегия: что можно
передавать, как сериализовать, какие ограничения.

### Исследование (индустриальная практика)

Анализ Astro, Qwik, Next.js, Fresh и других островных фреймворков (подтверждено
islands-architecture.com, docs.astro.build):

1. **Граница — JSON.stringify/JSON.parse.** Всё, что не переживает этот цикл
   (функции, `undefined`, `Symbol`, DOM-узлы, циклические ссылки), не может
   пересечь границу.
2. **Стандартный приём — tagged replacer/reviver** для специальных типов (Date → `{__type:'Date', iso:...}`, Map, Set, BigInt).
3. **Размер — критичен для TTI.** Рекомендации индустрии: < 50 KB на остров,
   JSON.parse занимает ~2–5ms на 10 KB.
4. **Payload встраивается в HTML** как содержимое `<script type="application/json">`
   (безопасно, не исполняется) или `data-props="..."`.
5. **Недетерминированные значения** (Date.now, Math.random) вызывают hydration-mismatch.
6. **Версионирование:** поле `__v` (version) позволяет client-side reconciliation.

### Решение

**Простые JSON-safe prop'ы + явное документирование границ. Без tagged-типов в MVP.**

Конкретно:
1. **Формат:** `data-props` с `JSON.stringify(props)`, внутри HTML-атрибута с
   корректным escaping (двойные кавычки → `&quot;`, `<` → `&lt;` и т.д.).
2. **Типы в MVP:**
   - Допустимы: `String`, `int`, `double`, `bool`, `null`, `List`, `Map<String, dynamic>`
     (с допустимыми значениями в листьях).
   - **Не** допустимы: функции, `DateTime` (использовать ISO-строку), `Uri` (строкой),
     `Color` (int-значением), циклические ссылки.
3. **Размер:** задокументированный лимит — «держите props минимальными». Валидатор
   devtools предупреждает при props > 10 KB (настраиваемый порог).
4. **Безопасность:**
   - На серверной стороне: `JsonEncoder` + экранирование специальных символов HTML.
   - На клиентской стороне: `JSON.parse` содержимого.
   - **Не** использовать eval / Function constructor / DOMParser.parseFromString для props.
5. **Расширение в будущем (Phase 8+):**
   - Опциональный tagged replacer/reviver для `DateTime`, `Uri`, `BigInt`.
   - Версионирование `__v` для оптимистичных обновлений.
6. **Клиентский код (hydraline_widgets):**
   - Island-загрузчик читает `dataset.props`, парсит JSON, пробрасывает как
     `initialData` в `addView({ hostElement, initialData })`.
   - Dart-сторона получает `Map<String, dynamic>` и должна сама десериализовать
     в нужные типы (предоставляется helper `IslandProps.of(view)`).

Обоснование:
- Соответствует индустриальным практикам (Astro, Qwik, Next.js).
- Простота MVP: JSON — универсальный контракт, понятный всем.
- Tagged types — преждевременное усложнение; большинству островов нужны строки и числа.
- Безопасность: встроенный механизм JSON-сериализации исключает инъекции (в отличие
  от сырых HTML-атрибутов без экранирования).

## Q7. Политика поддерживаемых версий Flutter

### Контекст

Зависимости от Flutter SDK: `hydraline_widgets`, `hydraline_build`, `hydraline_devtools`.
Нужно определить, какие версии и каналы поддерживаются.

### Данные о релевантных изменениях

- **Flutter 3.22:** стабилизирован multi-view API (`addView`/`removeView`, `ViewCollection`).
- **Flutter 3.38.8:** multi-view sizing работает корректно.
- **Flutter 3.41.x:** регрессия multi-view sizing (#185034) — canvas не совпадает
  с host-контейнером; PR'ы на исправление в процессе.
- **Flutter 3.44:** актуальный stable (2026-07), Skwasm для десктопов, CanvasKit
  для мобильных по умолчанию.
- Канал **beta**: новые фичи, может ломаться.
- Канал **master**: ежедневные изменения, непригоден для production-пакетов.

### Анализ

**Варианты политики:**
- **(A)** Только последний stable (Flutter ≥ 3.44).
- **(B)** Stable ≥ 3.22 (multi-view стабилизирован) и выше, с CI-матрицей.
- **(C)** Stable + beta (ждём фиксов и тестируем заранее).
- **(D)** «LTS»-окно: последние 3 stable-релиза.

**Факторы:**
- Hydraline — community package, не может требовать bleeding-edge.
- Multi-view — ключевая зависимость; регрессия #185034 критична.
- Пользователи не всегда на последней версии (корпоративные проекты, задержки
  обновления).
- CI должен тестировать на нескольких версиях для раннего обнаружения регрессий.

### Решение

**Политика «скользящего окна»: поддерживаем последние 3 стабильных релиза с
оговорками по известным регрессиям. CI на последнем + минимальном.**

Конкретно:
1. **Минимальная версия:** Flutter 3.22 (первый stable с multi-view). Объявляется
   как `>=3.22.0` в pubspec.
2. **Активная поддержка:** последние 3 стабильных релиза. На 2026-07 это 3.44, 3.41, 3.38.
3. **Известная регрессия:** Flutter 3.41.x — документируется как «поддерживается с
   оговоркой: multi-view sizing может быть неточным, рекомендуется 3.44+ или
   workaround (явные `viewConstraints` в `addView`)». CI тестирует на 3.41 с
   ожидаемыми warning'ами.
4. **CI-матрица:**
   - Минимальная: Flutter 3.22 (проверка «не развалилось ли»).
   - Текущий stable: Flutter 3.44 (полный набор тестов).
   - Будущий stable (beta): опционально, informational (не блокирует PR).
5. **Канал обновления:**
   - При выходе нового stable (напр. 3.45) — добавляется в матрицу, самая старая
     версия выбывает.
   - Решение принимается через issue «Add Flutter X.Y to CI matrix» с проверкой
     всех тестов.
6. **Публикация на pub.dev:** sdk constraint `>=3.22.0` без верхней границы
   (`<4.0.0` по умолчанию Dart).

Обоснование:
- Окно из 3 версий — стандарт для Dart/Flutter-сообщества (melos, go_router).
- 3.22 как минимум — доступ к multi-view API.
- 3.41 регрессия не блокирует поддержку, но документируется.
- CI-матрица даёт раннее обнаружение проблем без неподъёмного количества конфигураций.
- Не поддерживаем beta/master — это освобождает ресурсы под продуктовую разработку.
